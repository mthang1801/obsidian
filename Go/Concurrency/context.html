01

Khái Niệm & Triết Lý Thiết Kế

context.Context là một trong những interface quan trọng nhất trong Go, được giới thiệu từ Go 1.7. Nó giải quyết một bài toán cốt lõi trong lập trình concurrent và distributed systems: làm thế nào để truyền thông tin cancellation, deadline, và metadata xuyên suốt một call chain?

🎯 Context Giải Quyết Gì?

Trong một hệ thống phân tán, một HTTP request có thể fan-out thành hàng chục goroutines, gọi nhiều services, query nhiều DBs. Khi client disconnect hoặc timeout, tất cả phải dừng lại. Context là cơ chế để truyền tín hiệu "dừng lại" và metadata đó.

⚠️ Quy Tắc Bất Di Bất Dịch

Không bao giờ lưu Context trong struct. Luôn truyền Context như tham số đầu tiên của function, thường đặt tên là ctx context.Context. Đây là convention của cả Go standard library và ecosystem.

02

Context Interface — 4 Methods

context.Context chỉ là một interface với 4 methods. Sự đơn giản này là có chủ đích — Go luôn ưu tiên interface nhỏ ("The bigger the interface, the weaker the abstraction").

context/context.go — Standard Library Definition Go 
package context

// Context là interface cốt lõi — 4 methods, không hơn không kém.
// Mọi implementation đều phải satisfy interface này.
type Context interface {

    // Deadline trả về thời điểm context sẽ bị cancel tự động.
    // ok = false nếu context không có deadline.
    Deadline() (deadline time.Time, ok bool)

    // Done trả về một channel được ĐÓNG khi context bị cancel.
    // Nếu context không bao giờ bị cancel, Done() trả về nil.
    // Pattern: select { case <-ctx.Done(): return ctx.Err() }
    Done() <-chan struct{}

    // Err trả về lý do context bị cancel:
    // - context.Canceled: cancel() được gọi manually
    // - context.DeadlineExceeded: timeout/deadline hết hạn
    // - nil: context chưa bị cancel
    Err() error

    // Value trả về value gắn với key, hoặc nil nếu không có.
    // KEY RULE: dùng unexported type làm key để tránh collision.
    Value(key any) any
}

// Hai sentinel errors để check lý do cancel:
var (
    Canceled         = errors.New("context canceled")
    DeadlineExceeded error = deadlineExceededError{}
)

🔹 Done() — Channel-based Cancellation Signal

done_patterns.go — Cách sử dụng Done() đúng Go 
// Pattern 1: select trong goroutine loop — CHUẨN
func processJobs(ctx context.Context, jobs <-chan Job) error {
    for {
        select {
        case <-ctx.Done():
            return fmt.Errorf("processJobs stopped: %w", ctx.Err())
        case job, ok := <-jobs:
            if !ok {
                return nil // channel đóng bình thường
            }
            if err := processJob(ctx, job); err != nil {
                return err
            }
        }
    }
}

// Pattern 2: Polling check trong long-running ops
func heavyComputation(ctx context.Context, data []byte) (Result, error) {
    for i, chunk := range splitChunks(data) {
        // Check cancellation mỗi iteration — không blocking
        select {
        case <-ctx.Done():
            return Result{}, fmt.Errorf("cancelled at chunk %d: %w", i, ctx.Err())
        default:
        }
        processChunk(chunk)
    }
    return result, nil
}

// Pattern 3: Background.Done() == nil — không bao giờ bị cancel
func checkContextCancellable(ctx context.Context) bool {
    return ctx.Done() != nil // false với Background/TODO
}

🔹 Err() — Distinguishing Cancellation Reasons

err_handling.go Go 
func handleContextError(ctx context.Context) {
    switch ctx.Err() {
    case nil:
        // Context vẫn active — không làm gì
    case context.Canceled:
        // cancel() được gọi thủ công: client disconnect, user cancelled
        log.Info("request was cancelled by caller")
    case context.DeadlineExceeded:
        // Timeout/deadline hết hạn: trả về 504 Gateway Timeout
        log.Warn("request timed out — consider increasing timeout or optimizing")
    }
}

Context Constructors — Root & Derived

Minh họa: Context cancellation tree — khi cancel() được gọi, tín hiệu truyền xuống tất cả child goroutines qua ctx.Done()

Root Contexts (không có parent)

root_contexts.go Go 
// context.Background() — PRODUCTION CODE
// Dùng ở điểm khởi đầu: main, server, test setup
// Done() == nil, không bao giờ bị cancel
ctx := context.Background()

// context.TODO() — PLACEHOLDER
// Dùng khi chưa biết context nào phù hợp
// Dấu hiệu cần refactor — không dùng production lâu dài
ctx := context.TODO()

// Difference: Semantically different, functionally identical
// TODO → "I need to figure out the right context"
// Background → "This IS the root context"

Derived Contexts (có parent)

📐 Hierarchy Rule

Mọi derived context đều có parent. Khi parent bị cancel, toàn bộ subtree bị cancel tự động. Đây là cơ chế propagation.

04

WithCancel — Manual Cancellation

context.WithCancel trả về một context mới và một function cancel(). Khi cancel() được gọi, context bị cancel ngay lập tức, và tín hiệu propagate xuống toàn bộ children contexts.

func WithCancel(parent Context) (ctx Context, cancel CancelFunc)
// CancelFunc = func() — PHẢI gọi để tránh goroutine leak!
🚨 Critical: Luôn defer cancel()

Nếu không gọi cancel(), context và tất cả resources liên quan (goroutines, channels) sẽ bị leak cho đến khi parent bị cancel. Đây là bug phổ biến nhất với context.

Example 1: Basic WithCancel

with_cancel_basic.go Go 
package main

import (
    "context"
    "fmt"
    "time"
)

func main() {
    // Tạo cancellable context
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel() // LUÔN defer — tránh resource leak dù có panic

    // Spawn goroutine lắng nghe cancellation
    go func() {
        select {
        case <-ctx.Done():
            fmt.Println("goroutine: received cancellation signal")
            fmt.Println("reason:", ctx.Err()) // context.Canceled
        }
    }()

    time.Sleep(100 * time.Millisecond)
    cancel() // Trigger cancellation
    time.Sleep(10 * time.Millisecond) // Chờ goroutine log
}
// Output: goroutine: received cancellation signal
//         reason: context canceled

Example 2: Cancel Propagation Tree

cancel_propagation.go — Cancellation lan truyền xuống subtree Go 
func demonstratePropagation() {
    root, rootCancel := context.WithCancel(context.Background())
    defer rootCancel()

    // Child contexts
    child1, cancel1 := context.WithCancel(root)
    defer cancel1()

    child2, cancel2 := context.WithCancel(root)
    defer cancel2()

    // Grandchild context
    grandchild, cancelGC := context.WithCancel(child1)
    defer cancelGC()

    var wg sync.WaitGroup
    for name, ctx := range map[string]context.Context{
        "child1": child1, "child2": child2, "grandchild": grandchild,
    } {
        wg.Add(1)
        go func(name string, ctx context.Context) {
            defer wg.Done()
            <-ctx.Done()
            fmt.Printf("%s cancelled: %v
", name, ctx.Err())
        }(name, ctx)
    }

    time.Sleep(50 * time.Millisecond)
    fmt.Println("Cancelling ROOT context...")
    rootCancel() // Cancel root → TẤT CẢ children và grandchildren bị cancel
    wg.Wait()
}
// Output (thứ tự có thể khác nhau — concurrent):
// child1 cancelled: context canceled
// child2 cancelled: context canceled
// grandchild cancelled: context canceled

Example 3: Graceful Shutdown với WithCancel

graceful_shutdown.go — Production Pattern Go 
func main() {
    ctx, cancel := context.WithCancel(context.Background())

    // Listen OS signals để graceful shutdown
    sigCh := make(chan os.Signal, 1)
    signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)

    go func() {
        sig := <-sigCh
        log.Printf("Received signal: %v. Initiating graceful shutdown...", sig)
        cancel() // Signal all goroutines to stop
    }()

    // Khởi động các services — truyền ctx
    var wg sync.WaitGroup
    wg.Add(3)
    go runHTTPServer(ctx, &wg)
    go runWorkerPool(ctx, &wg)
    go runMetricsCollector(ctx, &wg)

    // Chờ tất cả components shutdown sạch
    wg.Wait()
    log.Println("All components shut down. Goodbye.")
}

func runHTTPServer(ctx context.Context, wg *sync.WaitGroup) {
    defer wg.Done()
    srv := &http.Server{Addr: ":8080"}

    go func() {
        <-ctx.Done()
        // Graceful shutdown với 30s timeout
        shutdownCtx, sc := context.WithTimeout(context.Background(), 30*time.Second)
        defer sc()
        srv.Shutdown(shutdownCtx)
    }()

    if err := srv.ListenAndServe(); err != http.ErrServerClosed {
        log.Printf("HTTP server error: %v", err)
    }
}

WithTimeout & WithDeadline — Time-based Cancellation

Minh họa: Ba cách tạo context — WithTimeout (tương đối), WithDeadline (tuyệt đối), WithCancel (thủ công)
WithTimeout(parent, duration)

Tự động cancel sau duration tính từ lúc gọi.
ctx, cancel := context.WithTimeout(parent, 5*time.Second)

WithDeadline(parent, time.Time)

Tự động cancel tại một thời điểm tuyệt đối.
ctx, cancel := context.WithDeadline(parent, time.Now().Add(5*time.Second))

WithTimeout(parent, d) ≡ WithDeadline(parent, time.Now().Add(d))
// Hai cái này tương đương nhau — WithTimeout chỉ là syntactic sugar

Example 1: HTTP Client với Timeout

http_client_timeout.go Go 
type UserService struct {
    client    *http.Client
    baseURL   string
    timeout   time.Duration
}

func (s *UserService) GetUser(ctx context.Context, id string) (*User, error) {
    // Tạo timeout context — KHÔNG override context của caller!
    // Nếu caller đã có deadline ngắn hơn, deadline đó được dùng.
    reqCtx, cancel := context.WithTimeout(ctx, s.timeout)
    defer cancel()

    url := fmt.Sprintf("%s/users/%s", s.baseURL, id)
    req, err := http.NewRequestWithContext(reqCtx, http.MethodGet, url, nil)
    if err != nil {
        return nil, fmt.Errorf("create request: %w", err)
    }

    resp, err := s.client.Do(req)
    if err != nil {
        // Check nếu là do context cancelled/timeout
        if errors.Is(err, context.DeadlineExceeded) {
            return nil, fmt.Errorf("GetUser %s timed out after %v", id, s.timeout)
        }
        return nil, fmt.Errorf("do request: %w", err)
    }
    defer resp.Body.Close()

    var user User
    if err := json.NewDecoder(resp.Body).Decode(&user); err != nil {
        return nil, fmt.Errorf("decode response: %w", err)
    }
    return &user, nil
}

Example 2: Database Query với Timeout

db_timeout.go — sql.DB với context Go 
type OrderRepository struct {
    db *sql.DB
}

func (r *OrderRepository) FindByUserID(
    ctx context.Context,
    userID int64,
) ([]Order, error) {
    // DB query timeout — 3 giây là reasonable cho OLTP
    queryCtx, cancel := context.WithTimeout(ctx, 3*time.Second)
    defer cancel()

    const query = `
        SELECT id, user_id, total, status, created_at
        FROM orders
        WHERE user_id = $1 AND deleted_at IS NULL
        ORDER BY created_at DESC
        LIMIT 100
    `

    rows, err := r.db.QueryContext(queryCtx, query, userID)
    if err != nil {
        return nil, fmt.Errorf("FindByUserID %d: %w", userID, err)
    }
    defer rows.Close()

    var orders []Order
    for rows.Next() {
        var o Order
        if err := rows.Scan(&o.ID, &o.UserID, &o.Total, &o.Status, &o.CreatedAt); err != nil {
            return nil, fmt.Errorf("scan order: %w", err)
        }
        orders = append(orders, o)
    }
    return orders, rows.Err()
}

// WithDeadline — khi cần coordinate nhiều ops trong SLA
func processOrderWithSLA(ctx context.Context, orderID string) error {
    // SLA: toàn bộ order processing phải xong trong 500ms
    deadline := time.Now().Add(500 * time.Millisecond)
    ctx, cancel := context.WithDeadline(ctx, deadline)
    defer cancel()

    // Deadline tự động propagate qua tất cả sub-calls
    order, err := fetchOrder(ctx, orderID)  // dùng remaining time
    if err != nil { return err }

    inventory, err := checkInventory(ctx, order)  // dùng remaining time
    if err != nil { return err }

    return reserveItems(ctx, inventory)  // dùng remaining time
}

// Check remaining time để adaptive logic
func checkRemainingTime(ctx context.Context) {
    if deadline, ok := ctx.Deadline(); ok {
        remaining := time.Until(deadline)
        log.Printf("Remaining time: %v", remaining)

        if remaining < 100*time.Millisecond {
            // Không đủ thời gian — fail fast
            log.Warn("Insufficient time remaining, aborting")
        }
    }
}

WithValue — Request-Scoped Data

context.WithValue cho phép gắn request-scoped data vào context. Data này propagate qua toàn bộ call chain — lý tưởng cho trace IDs, auth tokens, request metadata.

⚠️ WithValue KHÔNG phải là function parameters thay thế

Chỉ dùng WithValue cho cross-cutting concerns (tracing, auth, logging). KHÔNG dùng để truyền business logic parameters như userID vào DB query — đó là việc của function parameters rõ ràng.

Key Type Safety — Unexported Type Pattern

context_keys.go — Best Practice Key Management Go 
package middleware

// PATTERN: Dùng unexported type làm key — tránh collision với other packages
// Nếu dùng string key ("requestID"), bất kỳ package nào cũng có thể ghi đè!

type contextKey string // unexported type

const (
    keyRequestID contextKey = "requestID"
    keyUserID    contextKey = "userID"
    keyTraceID   contextKey = "traceID"
    keyLogger    contextKey = "logger"
)

// CHUẨN: Typed accessor functions — không expose key ra ngoài package
func WithRequestID(ctx context.Context, requestID string) context.Context {
    return context.WithValue(ctx, keyRequestID, requestID)
}

func RequestIDFrom(ctx context.Context) (string, bool) {
    id, ok := ctx.Value(keyRequestID).(string) // type assertion an toàn
    return id, ok
}

func MustRequestID(ctx context.Context) string {
    id, ok := RequestIDFrom(ctx)
    if !ok {
        return "unknown" // graceful fallback, không panic
    }
    return id
}

Example: Request Context Middleware

request_middleware.go — HTTP Middleware injecting context values Go 
// RequestContextMiddleware inject trace/request metadata vào context
func RequestContextMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        ctx := r.Context()

        // 1. Request ID — dùng từ header hoặc generate mới
        requestID := r.Header.Get("X-Request-ID")
        if requestID == "" {
            requestID = uuid.New().String()
        }
        ctx = WithRequestID(ctx, requestID)

        // 2. Trace ID cho distributed tracing
        traceID := r.Header.Get("X-Trace-ID")
        ctx = WithTraceID(ctx, traceID)

        // 3. Logger với pre-filled fields
        logger := slog.With(
            "requestID", requestID,
            "traceID", traceID,
            "method", r.Method,
            "path", r.URL.Path,
        )
        ctx = WithLogger(ctx, logger)

        // Set response header cho client
        w.Header().Set("X-Request-ID", requestID)

        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

// Sử dụng trong handler — clean, không cần params truyền xuống
func (h *OrderHandler) CreateOrder(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()
    logger := LoggerFrom(ctx) // có requestID tự động

    order, err := h.service.CreateOrder(ctx, req)
    if err != nil {
        logger.Error("create order failed", "error", err)
        writeError(w, err)
        return
    }
    logger.Info("order created", "orderID", order.ID)
    writeJSON(w, http.StatusCreated, order)
}

Context trong HTTP Server & gRPC

HTTP Server — Context tích hợp sẵn

http_server_context.go Go 
// HTTP Server tự động cancel context khi:
// 1. Client disconnect (TCP connection đóng)
// 2. Request deadline vượt quá (nếu có)
// 3. Server shutdown được gọi

func longRunningHandler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context() // Đây đã là cancellable context

    // Thêm request timeout (giả sử không có global timeout middleware)
    ctx, cancel := context.WithTimeout(ctx, 10*time.Second)
    defer cancel()

    result := make(chan Result, 1)

    go func() {
        result <- expensiveComputation(ctx)
    }()

    select {
    case res := <-result:
        json.NewEncoder(w).Encode(res)
    case <-ctx.Done():
        switch ctx.Err() {
        case context.DeadlineExceeded:
            http.Error(w, "Request timeout", http.StatusGatewayTimeout)
        case context.Canceled:
            // Client đã disconnect — không cần write response
            log.Info("client disconnected")
        }
    }
}

// Timeout Middleware — production grade
func TimeoutMiddleware(timeout time.Duration) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            ctx, cancel := context.WithTimeout(r.Context(), timeout)
            defer cancel()
            next.ServeHTTP(w, r.WithContext(ctx))
        })
    }
}

gRPC — Context là First-class Citizen

// gRPC Server Interceptor — context-based auth & tracing
func UnaryAuthInterceptor(
    ctx context.Context,
    req interface{},
    info *grpc.UnaryServerInfo,
    handler grpc.UnaryHandler,
) (interface{}, error) {
    // Extract metadata từ gRPC context
    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Unauthenticated, "missing metadata")
    }

    tokens := md.Get("authorization")
    if len(tokens) == 0 {
        return nil, status.Error(codes.Unauthenticated, "missing auth token")
    }

    userClaims, err := validateToken(tokens[0])
    if err != nil {
        return nil, status.Error(codes.Unauthenticated, "invalid token")
    }

    // Inject claims vào context để downstream sử dụng
    ctx = WithUserClaims(ctx, userClaims)
    return handler(ctx, req)
}

// gRPC Client — propagate deadline & metadata
func (c *GRPCClient) GetOrder(ctx context.Context, id string) (*pb.Order, error) {
    // gRPC tự propagate context deadline sang remote service
    // Không cần thêm timeout nếu ctx đã có deadline

    // Append outgoing metadata
    ctx = metadata.AppendToOutgoingContext(ctx,
        "x-request-id", MustRequestID(ctx),
        "x-trace-id", MustTraceID(ctx),
    )

    return c.client.GetOrder(ctx, &pb.GetOrderRequest{Id: id})
}

errgroup — Fan-out với Error Propagation

errgroup_pattern.go — golang.org/x/sync/errgroup Go 
import "golang.org/x/sync/errgroup"

// DashboardData aggregates dữ liệu từ nhiều sources song song
type DashboardData struct {
    User     *User
    Orders   []Order
    Payments []Payment
    Balance  float64
}

func (s *DashboardService) GetDashboard(
    ctx context.Context,
    userID int64,
) (*DashboardData, error) {
    // errgroup.WithContext: nếu 1 goroutine lỗi → cancel all others
    g, ctx := errgroup.WithContext(ctx)

    var (
        user     *User
        orders   []Order
        payments []Payment
        balance  float64
    )

    g.Go(func() (error) {
        var err error
        user, err = s.userRepo.FindByID(ctx, userID)
        return fmt.Errorf("fetch user: %w", err)
    })

    g.Go(func() error {
        var err error
        orders, err = s.orderRepo.FindByUserID(ctx, userID)
        return fmt.Errorf("fetch orders: %w", err)
    })

    g.Go(func() error {
        var err error
        payments, err = s.paymentClient.GetPayments(ctx, userID)
        return fmt.Errorf("fetch payments: %w", err)
    })

    g.Go(func() error {
        var err error
        balance, err = s.walletClient.GetBalance(ctx, userID)
        return fmt.Errorf("fetch balance: %w", err)
    })

    // Wait chờ tất cả hoặc trả về lỗi đầu tiên
    if err := g.Wait(); err != nil {
        return nil, fmt.Errorf("GetDashboard: %w", err)
    }

    return &DashboardData{
        User: user, Orders: orders,
        Payments: payments, Balance: balance,
    }, nil
}

Custom Context Implementation

custom_context.go — Implement Context interface (Advanced) Go 
// ReadOnlyContext — wrap context để ngăn cancel propagation
// Usecase: muốn truyền values nhưng không muốn deadline/cancel từ caller
type ReadOnlyContext struct {
    parent context.Context
}

func NewReadOnlyContext(parent context.Context) context.Context {
    return &ReadOnlyContext{parent: parent}
}

// Delegate Deadline/Done/Err cho background — không bị cancel
func (c *ReadOnlyContext) Deadline() (time.Time, bool) { return time.Time{}, false }
func (c *ReadOnlyContext) Done() <-chan struct{}           { return nil }
func (c *ReadOnlyContext) Err() error                       { return nil }

// Delegate Value — kế thừa toàn bộ values từ parent
func (c *ReadOnlyContext) Value(key any) any { return c.parent.Value(key) }

// Usecase: background job dùng values từ request context
// nhưng không muốn bị cancel khi request kết thúc
func ScheduleAsyncJob(requestCtx context.Context, payload Payload) {
    // Detach khỏi request lifecycle nhưng giữ trace/user info
    jobCtx := NewReadOnlyContext(requestCtx)

    go func() {
        // jobCtx.Done() == nil — không bị cancel khi request done
        // Nhưng vẫn có requestID, userID từ parent values
        jobCtx2, cancel := context.WithTimeout(jobCtx, 5*time.Minute)
        defer cancel()
        processJob(jobCtx2, payload)
    }()
}

Anti-patterns — Những Lỗi Thường Gặp

  • 🚫
    Lưu Context trong struct — Context phải luôn là tham số, không phải field. Struct context gây ra race conditions và lifecycle confusion.
  • 🚫
    Không gọi cancel() — Goroutine leak! Luôn defer cancel() ngay sau khi tạo cancellable context.
  • 🚫
    Dùng string/int làm key WithValue — Dễ bị override bởi package khác. Luôn dùng unexported custom type.
  • 🚫
    Truyền nil context — Sẽ panic! Dùng context.Background() hoặc context.TODO() nếu chưa có context phù hợp.
  • 🚫
    Dùng WithValue cho business parameters — Function parameters rõ ràng, không phải context values ẩn. Context values chỉ cho cross-cutting concerns.
  • 🚫
    Tạo context trong goroutine mà không truyền parent — Mất cancel propagation. Luôn truyền context từ caller xuống goroutine.
  • 🚫
    Bỏ qua ctx.Err() sau Done() — Không biết lý do cancel là Canceled hay DeadlineExceeded, không thể xử lý đúng cách.
❌ Anti-pattern: Context trong Struct
type BadService struct {
    ctx context.Context // ❌ KHÔNG làm thế này
    db  *sql.DB
}

func (s *BadService) GetUser(id int) (*User, error) {
    // ❌ dùng s.ctx — lifecycle không rõ ràng
    return s.db.QueryRowContext(s.ctx, ...)
}
✅ Correct: Context là tham số
type GoodService struct {
    db *sql.DB // ✅ không có ctx
}

func (s *GoodService) GetUser(
    ctx context.Context, // ✅ explicit param
    id int,
) (*User, error) {
    return s.db.QueryRowContext(ctx, ...)
}
❌ Anti-pattern: String Key
// ❌ string key — any package có thể overwrite
ctx = context.WithValue(ctx, "userID", 42)
ctx = context.WithValue(ctx, "userID", "hacked") // override!
✅ Correct: Unexported Type Key
type ctxKey string
const keyUserID ctxKey = "userID"

// ✅ type-safe — other packages cannot collide
ctx = context.WithValue(ctx, keyUserID, 42)

Expert Level — Context Internals & Performance

Cách Context được Implement Nội Bộ

🔬 Internal Implementation

Mỗi lần gọi WithCancel/WithTimeout/WithValue, Go tạo một struct nhỏ mới bao bọc parent context. Context tree là một linked list ngược (leaf → root). ctx.Value(key) thực hiện linear search từ leaf đến root — O(n) với n là độ sâu.

context_internals.go — Hiểu cách hoạt động bên trong Go 
// Simplified internal representation của context package
// (không phải code thực — chỉ để minh họa)

// cancelCtx — dùng cho WithCancel
type cancelCtx struct {
    parent   Context          // pointer lên parent
    mu       sync.Mutex       // protect done + children
    done     chan struct{}     // lazy-initialized
    children map[*cancelCtx]struct{} // children để propagate
    err      error
}

// valueCtx — dùng cho WithValue (chỉ 3 fields!)
type valueCtx struct {
    parent Context
    key    any
    val    any
}

// Value() lookup — linear search lên parent chain
func (c *valueCtx) Value(key any) any {
    if c.key == key {
        return c.val // found at this level
    }
    return c.parent.Value(key) // recurse up
}

// PERFORMANCE IMPLICATION: Nếu chain có 100 WithValue nodes,
// mỗi Value() lookup là O(100). Giải pháp:

// Pattern: Bundle nhiều values vào một struct, WithValue 1 lần
type RequestMetadata struct {
    RequestID string
    TraceID   string
    UserID    int64
    TenantID  string
}

type metaKey struct{}

func WithMetadata(ctx context.Context, meta RequestMetadata) context.Context {
    return context.WithValue(ctx, metaKey{}, meta) // 1 node thay vì 4
}

func MetadataFrom(ctx context.Context) (RequestMetadata, bool) {
    meta, ok := ctx.Value(metaKey{}).(RequestMetadata)
    return meta, ok
}

Context Deadline Budget — Budget Propagation Pattern

deadline_budget.go — Expert: Remaining budget-aware calls Go 
// DeadlineBudget — chia deadline budget cho nhiều sequential ops
type DeadlineBudget struct {
    ctx       context.Context
    remaining time.Duration
    allocated time.Duration
}

func NewDeadlineBudget(ctx context.Context) *DeadlineBudget {
    b := &DeadlineBudget{ctx: ctx}
    if deadline, ok := ctx.Deadline(); ok {
        b.remaining = time.Until(deadline)
    }
    return b
}

// Allocate trả về context với fraction của budget còn lại
func (b *DeadlineBudget) Allocate(fraction float64) (context.Context, context.CancelFunc) {
    if b.remaining == 0 {
        return b.ctx, func() {}
    }
    alloc := time.Duration(float64(b.remaining) * fraction)
    b.remaining -= alloc
    return context.WithTimeout(b.ctx, alloc)
}

// Usage: processOrder với 3 phases, chia budget hợp lý
func processOrderWithBudget(ctx context.Context, order Order) error {
    budget := NewDeadlineBudget(ctx)

    // Phase 1: 30% budget cho validation
    valCtx, valCancel := budget.Allocate(0.3)
    defer valCancel()
    if err := validateOrder(valCtx, order); err != nil { return err }

    // Phase 2: 40% budget cho inventory check
    invCtx, invCancel := budget.Allocate(0.4)
    defer invCancel()
    if err := checkInventory(invCtx, order); err != nil { return err }

    // Phase 3: 30% còn lại cho payment processing
    payCtx, payCancel := budget.Allocate(1.0)
    defer payCancel()
    return processPayment(payCtx, order)
}

Rules — 10 Quy Tắc Vàng

  • ✅
    Luôn ctx là tham số đầu tiên — func Foo(ctx context.Context, ...) — đây là convention toàn ecosystem.
  • ✅
    defer cancel() ngay sau WithCancel/WithTimeout — không có ngoại lệ. Ngay cả khi cancel sẽ được gọi sau, defer đảm bảo cleanup.
  • ✅
    Đừng override deadline của caller — WithTimeout(ctx, d) sẽ tự chọn deadline ngắn hơn giữa d và deadline hiện tại của ctx.
  • ✅
    Wrap context errors khi propagate — fmt.Errorf("operation: %w", ctx.Err()) để caller biết context hay business logic lỗi.
  • ✅
    Bundle values thay vì nhiều WithValue — O(n) lookup, nên gom tất cả request metadata vào một struct.
  • ✅
    Dùng unexported type cho key — type ctxKey struct{} — struct empty type là idiomatic nhất vì không cần constant.
  • ✅
    select luôn có default hoặc ctx.Done() — mọi goroutine long-running phải có exit path qua context.
  • ✅
    Detach context cho async jobs — Dùng context.Background() mới với values từ parent khi job cần sống lâu hơn request.
  • ✅
    Test với pre-cancelled context — Đảm bảo mọi function xử lý đúng khi nhận context đã cancel.
  • ✅
    Dùng errgroup thay vì sync.WaitGroup cho fan-out — errgroup tự động cancel context khi có lỗi đầu tiên.
