01

Pipeline là gì?

Trong Go, Pipeline là một chuỗi các stage (giai đoạn xử lý) kết nối với nhau thông qua channel. Mỗi stage nhận dữ liệu từ channel đầu vào, xử lý, rồi gửi kết quả sang channel đầu ra. Các stage chạy concurrently — đây là điểm khác biệt cốt lõi so với xử lý tuần tự.

💡

Triết lý Go: "Don't communicate by sharing memory; share memory by communicating." — Pipeline là hiện thân hoàn hảo của triết lý này, mọi dữ liệu chảy qua channel.

Hình 1.1: Sơ đồ kiến trúc Multi-Stage Pipeline trong Go Concurrency kết hợp cơ chế gửi dữ liệu qua các typed channels và hủy sớm an toàn qua shared context.Context
Pipeline Flow — Dữ liệu chảy qua các Stage

So sánh với Stream API (Java) và các ngôn ngữ khác

🎯

Khi nào dùng Pipeline Pattern? Khi có dữ liệu lớn cần xử lý theo từng bước, cần parallelism, hoặc muốn tách logic thành các unit độc lập, dễ test. Pipeline đặc biệt hiệu quả cho ETL, log processing, image processing, event streaming.

02

Cấu trúc một Stage

Mỗi stage trong pipeline Go tuân theo một template chuẩn: nhận <-chan T (input), trả về <-chan T (output), và chạy logic trong một goroutine riêng biệt.

Giải phẫu một Pipeline Stage
go // Template của một Pipeline Stage 
// Template chuẩn cho một stage
func stageTemplate(
    ctx context.Context,
    in <-chan InputType,   // nhận từ upstream
) <-chan OutputType {     // trả về cho downstream

    out := make(chan OutputType)

    go func() {
        defer close(out)          // ✅ Luôn close khi goroutine kết thúc

        for v := range in {       // ✅ range tự detect khi in bị close
            select {
            case <-ctx.Done():
                return            // ✅ Thoát khi có cancellation
            default:
            }

            result := process(v)  // logic xử lý

            select {
            case out <- result:    // gửi kết quả
            case <-ctx.Done():
                return
            }
        }
    }()

    return out
}
📥 Input: <-chan T

Receive-only channel. Stage chỉ đọc, không ghi. Đây là Go idiom để tách biệt producer/consumer.

📤 Output: chan<- T

Send-only channel được trả về. Caller nhận channel này và pipe vào stage tiếp theo.

🔒 Ownership

Stage tạo ra channel thì phải close() nó. Receiver không bao giờ close channel — đây là quy tắc bất biến.

⚡ Concurrency

Mỗi stage chạy trong goroutine riêng, do đó tất cả stages chạy song song — đây là toàn bộ "phép màu" của pipeline.

03

Pipeline cơ bản — Generator + Transform + Sink

Pipeline đơn giản nhất gồm 3 stage: Generator (tạo dữ liệu), Transform (xử lý), và Sink (tiêu thụ kết quả).

go basic_pipeline.go 
package main

import "fmt"

// ─── Stage 1: Generator ────────────────────────────────────────────
// Tạo dữ liệu từ slice và đẩy vào channel
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range nums {
            out <- n
        }
    }()
    return out
}

// ─── Stage 2: Transform — nhân đôi giá trị ────────────────────────
func double(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            out <- n * 2
        }
    }()
    return out
}

// ─── Stage 3: Transform — bình phương ─────────────────────────────
func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            out <- n * n
        }
    }()
    return out
}

// ─── Sink: tiêu thụ kết quả ────────────────────────────────────────
func main() {
    // Kết nối các stage: generate → double → square → print
    nums  := generate(1, 2, 3, 4, 5)
    doubled := double(nums)
    squared := square(doubled)

    // Sink: đọc kết quả cuối
    for result := range squared {
        fmt.Println(result) // 4, 16, 36, 64, 100
    }
}

/*
  Dữ liệu chảy:
  [1,2,3,4,5]  →(double)→  [2,4,6,8,10]  →(square)→  [4,16,36,64,100]
  
  Ba goroutine chạy ĐỒNG THỜI:
  - generate goroutine đang gửi 3
  - double goroutine đang xử lý 2, gửi 4
  - square goroutine đang nhận 4, tính 16
*/

Generator Pattern — Chuyển đổi các nguồn dữ liệu khác nhau

go generators.go — Các loại Generator phổ biến 
import (
    "bufio"
    "context"
    "database/sql"
    "os"
    "time"
)

// ── Generator từ Slice ────────────────────────────────────────────
func FromSlice[T any](ctx context.Context, items []T) <-chan T {
    out := make(chan T)
    go func() {
        defer close(out)
        for _, item := range items {
            select {
            case out <- item:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// ── Generator từ File (line by line) ─────────────────────────────
func FromFile(ctx context.Context, path string) (<-chan string, <-chan error) {
    out  := make(chan string)
    errc := make(chan error, 1)

    go func() {
        defer close(out)
        defer close(errc)

        f, err := os.Open(path)
        if err != nil {
            errc <- err
            return
        }
        defer f.Close()

        scanner := bufio.NewScanner(f)
        for scanner.Scan() {
            select {
            case out <- scanner.Text():
            case <-ctx.Done():
                return
            }
        }
        if err := scanner.Err(); err != nil {
            errc <- err
        }
    }()
    return out, errc
}

// ── Generator từ Database ─────────────────────────────────────────
func FromDB(ctx context.Context, db *sql.DB, query string) <-chan *sql.Rows {
    out := make(chan *sql.Rows, 1)
    go func() {
        defer close(out)
        rows, err := db.QueryContext(ctx, query)
        if err != nil {
            return
        }
        out <- rows
    }()
    return out
}

// ── Ticker Generator — emit mỗi N giây ───────────────────────────
func Ticker(ctx context.Context, interval time.Duration) <-chan time.Time {
    out := make(chan time.Time)
    go func() {
        defer close(out)
        t := time.NewTicker(interval)
        defer t.Stop()
        for {
            select {
            case tick := <-t.C:
                out <- tick
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

Multi-Stage Pipeline

Trong thực tế, pipeline thường gồm nhiều loại stage khác nhau: Transformation, Filter, Batch, và Merge. Mỗi stage độc lập và có thể test riêng lẻ.

Multi-Stage Pipeline — Xử lý Log Events

Transformation Stage

go transform_stage.go 
type LogEntry struct {
    Timestamp time.Time
    Level     string
    Message   string
    Fields    map[string]string
}

// Stage: Chuyển đổi raw string → LogEntry (struct)
func ParseLog(ctx context.Context, in <-chan string) <-chan LogEntry {
    out := make(chan LogEntry, 100) // buffer để tránh blocking
    go func() {
        defer close(out)
        for line := range in {
            entry, err := parseLogLine(line)
            if err != nil {
                continue // bỏ qua dòng lỗi (hoặc dùng error channel)
            }
            select {
            case out <- entry:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// Stage: Enrich — thêm thông tin bổ sung
func EnrichLog(ctx context.Context, in <-chan LogEntry, svc string) <-chan LogEntry {
    out := make(chan LogEntry, 100)
    go func() {
        defer close(out)
        for entry := range in {
            if entry.Fields == nil {
                entry.Fields = make(map[string]string)
            }
            entry.Fields["service"] = svc
            entry.Fields["host"], _ = os.Hostname()
            select {
            case out <- entry:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

Filter Stage

go filter_stage.go 
// Filter stage: chỉ cho qua các entry thỏa điều kiện
func FilterByLevel(ctx context.Context, in <-chan LogEntry, levels ...string) <-chan LogEntry {
    levelSet := make(map[string]struct{})
    for _, l := range levels {
        levelSet[l] = struct{}{}
    }

    out := make(chan LogEntry, 100)
    go func() {
        defer close(out)
        for entry := range in {
            if _, ok := levelSet[entry.Level]; !ok {
                continue // bỏ qua nếu không khớp
            }
            select {
            case out <- entry:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// Generic Filter — dùng được cho mọi kiểu
func Filter[T any](ctx context.Context, in <-chan T, pred func(T) bool) <-chan T {
    out := make(chan T, 64)
    go func() {
        defer close(out)
        for v := range in {
            if !pred(v) {
                continue
            }
            select {
            case out <- v:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// Stage: Batch — gom N items thành một slice
func Batch[T any](ctx context.Context, in <-chan T, size int, flush time.Duration) <-chan []T {
    out := make(chan []T)
    go func() {
        defer close(out)
        batch := make([]T, 0, size)
        ticker := time.NewTicker(flush)
        defer ticker.Stop()

        send := func() {
            if len(batch) == 0 { return }
            out <- batch
            batch = make([]T, 0, size)
        }

        for {
            select {
            case v, ok := <-in:
                if !ok {
                    send()      // flush batch cuối cùng
                    return
                }
                batch = append(batch, v)
                if len(batch) >= size {
                    send()
                }
            case <-ticker.C:
                send()          // time-based flush
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}
05

Fan-Out / Fan-In — Xử lý Song Song

Fan-Out: phân phối công việc từ 1 channel sang N goroutine. Fan-In: gộp kết quả từ N channel về 1. Kết hợp hai pattern này tạo ra khả năng xử lý song song mạnh mẽ.

Fan-Out / Fan-In Pattern
go fan_out_fan_in.go 
package main

import (
    "context"
    "sync"
)

// FanOut: phân phối items từ 1 channel vào N worker channels
func FanOut[T any](
    ctx context.Context,
    in <-chan T,
    n int,
    worker func(ctx context.Context, in <-chan T) <-chan T,
) []<-chan T {
    channels := make([]<-chan T, n)
    for i := range channels {
        channels[i] = worker(ctx, in) // N workers đọc cùng channel in
    }
    return channels
}

// FanIn (Merge): gộp N channels vào 1
func FanIn[T any](ctx context.Context, channels ...<-chan T) <-chan T {
    var wg sync.WaitGroup
    merged := make(chan T, len(channels))

    // forward: đọc từ một channel và gửi vào merged
    forward := func(ch <-chan T) {
        defer wg.Done()
        for v := range ch {
            select {
            case merged <- v:
            case <-ctx.Done():
                return
            }
        }
    }

    wg.Add(len(channels))
    for _, ch := range channels {
        go forward(ch)
    }

    // Close merged khi tất cả goroutines kết thúc
    go func() {
        wg.Wait()
        close(merged)
    }()

    return merged
}

// ── Ví dụ: xử lý URL song song ────────────────────────────────────
type Result struct {
    URL    string
    Status int
    Err    error
}

func fetchWorker(ctx context.Context, urls <-chan string) <-chan Result {
    out := make(chan Result)
    go func() {
        defer close(out)
        for url := range urls {
            resp, err := http.Get(url)
            result := Result{URL: url, Err: err}
            if err == nil {
                result.Status = resp.StatusCode
                resp.Body.Close()
            }
            select {
            case out <- result:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()

    urls := FromSlice(ctx, []string{
        "https://api1.example.com",
        "https://api2.example.com",
        // ...100 URLs
    })

    // Fan-out: 5 workers song song
    workers := FanOut(ctx, urls, 5, func(ctx context.Context, in <-chan string) <-chan Result {
        return fetchWorker(ctx, in)
    })

    // Fan-in: gộp kết quả
    results := FanIn(ctx, workers...)

    for r := range results {
        if r.Err != nil {
            log.Printf("Error %s: %v", r.URL, r.Err)
            continue
        }
        fmt.Printf("%s → %d\n", r.URL, r.Status)
    }
}
06

Context & Cancellation — Pipeline An Toàn

Một pipeline production-ready phải hỗ trợ graceful cancellation. Nếu bất kỳ stage nào gặp lỗi, toàn bộ pipeline nên dừng lại sạch sẽ mà không leak goroutine.

⚠️

Goroutine Leak! Nếu stage không check ctx.Done(), khi pipeline bị cancel, goroutine sẽ bị stuck mãi mãi tại out <- value vì không ai đọc channel nữa.

go pipeline_with_context.go — Pipeline an toàn với Context 
package main

import (
    "context"
    "fmt"
    "time"
)

// WithCancel: bọc context vào pipeline, tự cancel khi có lỗi
type Pipeline[T any] struct {
    ctx    context.Context
    cancel context.CancelFunc
}

func NewPipeline[T any](parent context.Context) *Pipeline[T] {
    ctx, cancel := context.WithCancel(parent)
    return &Pipeline[T]{ctx: ctx, cancel: cancel}
}

// ── Stage có thể cancel toàn bộ pipeline khi gặp lỗi ──────────────
func ValidateStage(
    ctx context.Context,
    cancel context.CancelFunc,
    in <-chan Record,
) <-chan Record {
    out := make(chan Record)
    go func() {
        defer close(out)
        for rec := range in {
            if err := rec.Validate(); err != nil {
                cancel() // ← cancel toàn bộ pipeline!
                return
            }
            select {
            case out <- rec:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// ── Pipeline với timeout ───────────────────────────────────────────
func RunWithTimeout(timeout time.Duration) error {
    ctx, cancel := context.WithTimeout(context.Background(), timeout)
    defer cancel()

    // Khởi tạo pipeline
    source := generateData(ctx)
    stage1 := transform(ctx, source)
    stage2 := filter(ctx, stage1)
    
    // Sink: đọc cho đến khi xong hoặc timeout
    for {
        select {
        case v, ok := <-stage2:
            if !ok {
                return nil // pipeline hoàn thành
            }
            fmt.Println(v)
        case <-ctx.Done():
            return ctx.Err() // timeout hoặc cancel
        }
    }
}

Error Handling trong Pipeline

Go không có exception. Trong pipeline, lỗi được truyền qua error channel riêng biệt hoặc bọc vào một Result type chứa cả value lẫn error.

Pattern 1: Result<T> — Bọc error vào kết quả

go error_handling_pipeline.go 
// Result bọc value + error — tương tự Result<T, E> trong Rust
type Result[T any] struct {
    Value T
    Err   error
}

func Ok[T any](v T) Result[T]    { return Result[T]{Value: v} }
func Err[T any](err error) Result[T] { return Result[T]{Err: err} }

// Stage với error handling — mọi lỗi đều được truyền xuống
func ParseStage(ctx context.Context, in <-chan string) <-chan Result[Record] {
    out := make(chan Result[Record], 64)
    go func() {
        defer close(out)
        for line := range in {
            rec, err := parse(line)
            var result Result[Record]
            if err != nil {
                result = Err[Record](fmt.Errorf("parse %q: %w", line, err))
            } else {
                result = Ok(rec)
            }
            select {
            case out <- result:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// Sink: tách thành công và thất bại
func CollectResults[T any](results <-chan Result[T]) ([]T, []error) {
    var values []T
    var errors []error

    for r := range results {
        if r.Err != nil {
            errors = append(errors, r.Err)
        } else {
            values = append(values, r.Value)
        }
    }
    return values, errors
}

// Pattern 2: errgroup — dừng ngay khi có lỗi đầu tiên
func RunPipelineErrGroup(ctx context.Context, data []string) error {
    g, ctx := errgroup.WithContext(ctx)
    
    source := FromSlice(ctx, data)
    
    g.Go(func() error {
        for rec := range source {
            if err := process(rec); err != nil {
                return err // tự động cancel ctx
            }
        }
        return nil
    })
    
    return g.Wait()
}
08

Generic Pipeline (Go 1.18+)

⬡ Advanced

Go 1.18 giới thiệu Generics, cho phép viết các stage/operator tái sử dụng được cho mọi kiểu dữ liệu — giống Stream API trong Java.

go generic_pipeline.go — Stream API cho Go 
package pipeline

import (
    "context"
    "sync"
)

// ── Core generic operators ─────────────────────────────────────────

// Map: transform mỗi element — giống Stream.map() Java
func Map[In, Out any](
    ctx context.Context,
    in <-chan In,
    f func(In) Out,
) <-chan Out {
    out := make(chan Out, cap(in))
    go func() {
        defer close(out)
        for v := range in {
            select {
            case out <- f(v):
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// FlatMap: map + flatten — giống Stream.flatMap() Java
func FlatMap[In, Out any](
    ctx context.Context,
    in <-chan In,
    f func(In) []Out,
) <-chan Out {
    out := make(chan Out, 64)
    go func() {
        defer close(out)
        for v := range in {
            for _, item := range f(v) {
                select {
                case out <- item:
                case <-ctx.Done():
                    return
                }
            }
        }
    }()
    return out
}

// Reduce: tổng hợp — giống Stream.reduce() Java
func Reduce[T, Acc any](
    ctx context.Context,
    in <-chan T,
    init Acc,
    f func(Acc, T) Acc,
) Acc {
    acc := init
    for {
        select {
        case v, ok := <-in:
            if !ok {
                return acc
            }
            acc = f(acc, v)
        case <-ctx.Done():
            return acc
        }
    }
}

// Take: lấy N items đầu tiên
func Take[T any](ctx context.Context, in <-chan T, n int) <-chan T {
    out := make(chan T)
    go func() {
        defer close(out)
        count := 0
        for v := range in {
            if count >= n { return }
            select {
            case out <- v:
                count++
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

// Parallel Map: chạy f song song với N goroutines
func ParallelMap[In, Out any](
    ctx context.Context,
    in <-chan In,
    workers int,
    f func(In) Out,
) <-chan Out {
    // Fan-out
    channels := make([]<-chan Out, workers)
    for i := range channels {
        channels[i] = Map(ctx, in, f)
    }
    // Fan-in
    return FanIn(ctx, channels...)
}

// ── Builder Pattern: fluent API ───────────────────────────────────
type Stream[T any] struct {
    ctx context.Context
    ch  <-chan T
}

func NewStream[T any](ctx context.Context, ch <-chan T) *Stream[T] {
    return &Stream[T]{ctx: ctx, ch: ch}
}

func (s *Stream[T]) Filter(pred func(T) bool) *Stream[T] {
    return &Stream[T]{ctx: s.ctx, ch: Filter(s.ctx, s.ch, pred)}
}

func (s *Stream[T]) Take(n int) *Stream[T] {
    return &Stream[T]{ctx: s.ctx, ch: Take(s.ctx, s.ch, n)}
}

func (s *Stream[T]) Collect() []T {
    var result []T
    for v := range s.ch {
        result = append(result, v)
    }
    return result
}

// ── Usage: giống Stream API trong Java ────────────────────────────

Functional Composition — Ghép Pipeline tự động

go functional_composition.go 
// Stage là một function type
type Stage[T any] func(ctx context.Context, in <-chan T) <-chan T

// Compose: kết nối nhiều stages thành một pipeline
func Compose[T any](stages ...Stage[T]) Stage[T] {
    return func(ctx context.Context, in <-chan T) <-chan T {
        current := in
        for _, stage := range stages {
            current = stage(ctx, current)
        }
        return current
    }
}

// Ví dụ sử dụng Compose
func BuildLogPipeline() Stage[string] {
    return Compose(
        func(ctx context.Context, in <-chan string) <-chan string {
            return trimStage(ctx, in)
        },
        func(ctx context.Context, in <-chan string) <-chan string {
            return dedupeStage(ctx, in)
        },
        func(ctx context.Context, in <-chan string) <-chan string {
            return normalizeStage(ctx, in)
        },
    )
}

func main() {
    ctx := context.Background()
    source := FromFile(ctx, "access.log")
    pipeline := BuildLogPipeline()
    result := pipeline(ctx, source)
    for line := range result {
        fmt.Println(line)
    }
}

Expert — Lock-Free & Ring Buffer Pipeline

★ Expert

Khi channel overhead trở thành bottleneck (~50-100ns/op), cần dùng các kỹ thuật lock-free với atomic operations và Ring Buffer (SPSC - Single Producer Single Consumer).

🔬

Khi nào cần Lock-Free Pipeline? Khi đo benchmark và thấy channel là bottleneck (thường >10M items/s). Đừng tối ưu sớm — hãy đo trước. Channel thông thường đủ dùng cho hầu hết các trường hợp.

go lockfree_ringbuffer.go — SPSC Ring Buffer 
package lockfree

import (
    "runtime"
    "sync/atomic"
    "unsafe"
)

// RingBuffer: SPSC (Single Producer, Single Consumer) lock-free queue
// Sử dụng atomic operations thay cho mutex hay channel
type RingBuffer[T any] struct {
    _     [64]byte          // padding tránh false sharing
    head  atomic.Uint64     // index producer ghi vào
    _     [56]byte
    tail  atomic.Uint64     // index consumer đọc từ
    _     [56]byte
    mask  uint64             // size - 1 (size phải là power-of-2)
    slots []slot[T]
}

type slot[T any] struct {
    seq   atomic.Uint64
    value T
}

func NewRingBuffer[T any](size uint64) *RingBuffer[T] {
    // size phải là power of 2
    if size & (size-1) != 0 {
        panic("size phải là power of 2")
    }
    rb := &RingBuffer[T]{
        mask:  size - 1,
        slots: make([]slot[T], size),
    }
    for i := range rb.slots {
        rb.slots[i].seq.Store(uint64(i))
    }
    return rb
}

// Enqueue: producer ghi vào (không block, return false nếu full)
func (rb *RingBuffer[T]) Enqueue(val T) bool {
    var pos uint64
    for {
        pos = rb.head.Load()
        s := &rb.slots[pos&rb.mask]
        seq := s.seq.Load()
        diff := int64(seq) - int64(pos)
        if diff == 0 {
            if rb.head.CompareAndSwap(pos, pos+1) {
                s.value = val
                s.seq.Store(pos + 1)
                return true
            }
        } else if diff < 0 {
            return false // buffer full
        } else {
            runtime.Gosched() // nhường CPU
        }
    }
}

// Dequeue: consumer đọc ra (return false nếu empty)
func (rb *RingBuffer[T]) Dequeue() (T, bool) {
    var pos uint64
    for {
        pos = rb.tail.Load()
        s := &rb.slots[pos&rb.mask]
        seq := s.seq.Load()
        diff := int64(seq) - int64(pos+1)
        if diff == 0 {
            if rb.tail.CompareAndSwap(pos, pos+1) {
                val := s.value
                var zero T
                s.value = zero           // giải phóng reference (GC)
                s.seq.Store(pos + rb.mask + 1)
                return val, true
            }
        } else if diff < 0 {
            var zero T
            return zero, false // buffer empty
        } else {
            runtime.Gosched()
        }
    }
}

// ── Benchmark comparison ──────────────────────────────────────────
/*
  BenchmarkChannel/unbuffered    ~150 ns/op
  BenchmarkChannel/buffered-1024 ~40  ns/op
  BenchmarkRingBuffer/spsc-1024  ~8   ns/op   ← ~5x nhanh hơn buffered chan
  
  Kết luận: chỉ dùng Ring Buffer khi đã profile và xác định channel là bottleneck
*/

Real-World: Hệ thống ETL Pipeline hoàn chỉnh

★ Expert

Kết hợp tất cả các pattern để xây dựng một hệ thống ETL (Extract-Transform-Load) thực tế: đọc từ nhiều nguồn, xử lý song song, ghi vào database với retry logic.

ETL Pipeline Architecture
go etl_pipeline.go — Hệ thống ETL hoàn chỉnh 
package etl

import (
    "context"
    "database/sql"
    "fmt"
    "sync"
    "time"
)

type Record struct {
    ID        string
    Source    string
    Data      map[string]any
    Timestamp time.Time
}

type ETLConfig struct {
    Workers     int
    BatchSize   int
    FlushPeriod time.Duration
    MaxRetries  int
    DB          *sql.DB
}

type ETLPipeline struct {
    config ETLConfig
    errc   chan error
    done   chan struct{}
    wg     sync.WaitGroup
}

func New(cfg ETLConfig) *ETLPipeline {
    return &ETLPipeline{
        config: cfg,
        errc:   make(chan error, 100),
        done:   make(chan struct{}),
    }
}

// Run: khởi chạy ETL pipeline hoàn chỉnh
func (p *ETLPipeline) Run(ctx context.Context, sources ...Source) error {
    ctx, cancel := context.WithCancel(ctx)
    defer cancel()

    // ── EXTRACT: đọc từ nhiều nguồn ──────────────────────────────
    extractChannels := make([]<-chan Record, len(sources))
    for i, src := range sources {
        extractChannels[i] = src.Extract(ctx)
    }
    merged := FanIn(ctx, extractChannels...)

    // ── TRANSFORM: xử lý song song ───────────────────────────────
    validated := ParallelMap(ctx, merged, p.config.Workers, func(r Record) Record {
        return p.validate(r)
    })

    normalized := Map(ctx, validated, func(r Record) Record {
        return p.normalize(r)
    })

    enriched := Map(ctx, normalized, func(r Record) Record {
        enriched, err := p.enrich(ctx, r)
        if err != nil {
            p.errc <- fmt.Errorf("enrich %s: %w", r.ID, err)
        }
        return enriched
    })

    // ── LOAD: batch + retry ───────────────────────────────────────
    batches := Batch(ctx, enriched, p.config.BatchSize, p.config.FlushPeriod)

    for batch := range batches {
        batch := batch
        p.wg.Add(1)
        go func() {
            defer p.wg.Done()
            if err := p.loadWithRetry(ctx, batch); err != nil {
                p.errc <- err
            }
        }()
    }

    p.wg.Wait()
    close(p.errc)
    close(p.done)
    return nil
}

// loadWithRetry: ghi với exponential backoff retry
func (p *ETLPipeline) loadWithRetry(ctx context.Context, batch []Record) error {
    backoff := 100 * time.Millisecond
    var lastErr error

    for attempt := 0; attempt < p.config.MaxRetries; attempt++ {
        if err := p.bulkInsert(ctx, batch); err == nil {
            return nil // thành công
        } else {
            lastErr = err
        }

        select {
        case <-time.After(backoff):
            backoff *= 2 // exponential backoff
        case <-ctx.Done():
            return ctx.Err()
        }
    }
    return fmt.Errorf("sau %d lần thử: %w", p.config.MaxRetries, lastErr)
}

// Metrics: theo dõi hiệu suất pipeline
type Metrics struct {
    Processed  atomic.Int64
    Errors     atomic.Int64
    Latency    atomic.Int64  // nanoseconds avg
    Throughput atomic.Int64  // items/sec
}

// Usage example
func ExampleETL() {
    cfg := ETLConfig{
        Workers:     runtime.NumCPU() * 2,
        BatchSize:   1000,
        FlushPeriod: 5 * time.Second,
        MaxRetries:  3,
    }

    pipeline := New(cfg)
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    err := pipeline.Run(ctx,
        &CSVSource{Path: "data/*.csv"},
        &APISource{URL: "https://api.example.com/data"},
        &KafkaSource{Topic: "events"},
    )
    if err != nil {
        log.Fatal(err)
    }
}

Performance Tips & Benchmarks

✅ Buffer Channel đúng cỡ

Buffer size = số lượng items mà stage nhanh nhất có thể produce trước khi stage chậm kịp xử lý. Thường = worker count × 2.

✅ Batch nhỏ thay vì item đơn lẻ

Gửi []T thay vì từng T một khi có thể. Giảm số lần channel operation, tăng throughput đáng kể.

✅ Tránh Allocation trong hot path

Dùng sync.Pool cho objects thường xuyên được tạo/hủy trong pipeline (như buffer, struct tạm thời).

✅ GOMAXPROCS = CPU cores

Fan-out worker count nên = runtime.NumCPU() cho CPU-bound tasks, hoặc nhiều hơn cho I/O-bound tasks.

✅ False Sharing

Thêm padding giữa atomic counters trên các CPU cache lines khác nhau (cache line = 64 bytes).

⚠️ Đừng over-parallelize

Quá nhiều goroutine gây context-switch overhead. Đo trước với go test -bench -cpuprofile.
