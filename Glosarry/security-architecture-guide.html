01
mTLS — Mutual TLS
Xác thực hai chiều giữa client và server bằng certificate
IDENTITY
💡 Định nghĩa
mTLS là gì?

Mutual TLS (mTLS) là phần mở rộng của giao thức TLS thông thường. Trong TLS thông thường, chỉ có server xuất trình certificate cho client xác minh. Với mTLS, cả hai phía — client và server — đều phải xuất trình certificate hợp lệ được ký bởi một Certificate Authority (CA) tin cậy chung. Đây là nền tảng của xác thực service-to-service trong kiến trúc microservices và Zero Trust.

mTLS giải quyết câu hỏi: "Tôi không chỉ biết đây là server đúng, mà còn biết đây là client được phép kết nối." Không cần username/password, không cần API key — certificate chính là danh tính.

TLS thông thường vs mTLS

TLS vs mTLS — Luồng xác thực so sánh

Các thành phần cốt lõi của mTLS

🏛️ Certificate Authority (CA)

Tổ chức/hệ thống ký certificate cho các service. Trong internal PKI (Public Key Infrastructure), tổ chức tự vận hành CA riêng (như CFSSL, Vault PKI, cert-manager). Cả client và server phải tin cùng một CA.

📜 X.509 Certificate

Chứa: Public key, thông tin định danh (Subject), CA chữ ký, thời hạn (NotBefore/NotAfter), Subject Alternative Name (SAN) dùng để verify hostname/service name.

🔑 Private Key

Không bao giờ rời khỏi service sở hữu. Dùng để tạo chữ ký chứng minh sở hữu certificate. Cần được bảo vệ nghiêm ngặt — nên lưu trong Secret Manager hoặc HSM, không bao giờ commit vào code.

Certificate Rotation — Thách thức lớn nhất

⚠️

Certificate expiry là nguồn gốc của nhiều outage nghiêm trọng. Hệ thống mTLS yêu cầu certificate rotation tự động. Tools như cert-manager (Kubernetes), Vault PKI với lease TTL ngắn (24h–7 ngày), hoặc SPIFFE/SPIRE giúp tự động hóa lifecycle quản lý certificate.

❌ Không nên làm
  • • Certificate thời hạn 1–2 năm, rotate thủ công
  • • Private key lưu trong file config hoặc environment variable
  • • Dùng self-signed cert mà không có CA hierarchy
  • • Bỏ qua verify hostname/SAN trong code
✅ Best Practices
  • • Certificate TTL ngắn (24h–7 ngày) + auto-rotate
  • • Private key trong Vault/HSM hoặc Kubernetes Secret (encrypted)
  • • Internal PKI với Root CA offline, Intermediate CA online
  • • SPIFFE/SPIRE cho workload identity trong multi-cluster

mTLS — Use Cases thực tế
Service mesh, API Gateway, Zero Trust network
USE CASES
Case 1.1 — Service Mesh: mTLS trong Kubernetes với Istio/LinkerdINTERMEDIATE
📌 Problem Statement

Một hệ thống microservices chạy trên Kubernetes gồm 20+ services giao tiếp với nhau qua HTTP. Traffic nội bộ hoàn toàn không được mã hóa — nếu một pod bị compromise, attacker có thể sniff traffic hoặc impersonate service khác. Security team yêu cầu mọi service-to-service communication phải được mã hóa và xác thực. Việc sửa code từng service để implement mTLS riêng lẻ sẽ tốn hàng tháng.

✅ Solution — Transparent mTLS qua Service Mesh Sidecar

Service Mesh (Istio hoặc Linkerd) inject một sidecar proxy (Envoy/Linkerd2-proxy) vào mỗi pod. Sidecar tự động xử lý mTLS: nhận certificate từ control plane, thực hiện handshake, mã hóa/giải mã traffic — hoàn toàn transparent với application code. Developer không cần sửa một dòng code nào.

📊 Visual — Architecture Diagram
📝 Note & Conclusion

Service mesh là cách tiếp cận đúng đắn khi cần roll out mTLS cho nhiều service: không cần sửa code, certificate lifecycle tự động, policy quản lý tập trung. Trade-off: overhead latency ~1ms per hop và memory/CPU cho sidecar (~50MB RAM/pod). Istio PeerAuthentication cho phép set STRICT mode (chỉ chấp nhận mTLS) hoặc PERMISSIVE (chấp nhận cả plain text, dùng trong giai đoạn migration). Với Linkerd, tất cả meshed traffic tự động là mTLS.

Case 1.2 — mTLS cho API Gateway: xác thực IoT devices và B2B partnersADVANCED
📌 Problem Statement

Một công ty fintech cần cho phép 500 thiết bị IoT (POS terminals) và 30 B2B partners gọi API. API key dễ bị đánh cắp, rotate khó. Yêu cầu: mỗi device và partner phải có danh tính không thể giả mạo, không phụ thuộc vào secret key truyền qua network. Khi một device bị compromise, cần revoke ngay lập tức mà không ảnh hưởng các device khác.

✅ Solution — Certificate-based Authentication tại API Gateway

Cấp certificate riêng cho từng device/partner. API Gateway (NGINX, Kong, AWS API Gateway) config verify client certificate. Certificate chứa device ID trong Subject/SAN. Khi device bị compromise, chỉ cần add cert vào CRL (Certificate Revocation List) hoặc OCSP — revoke tức thì mà không ảnh hưởng các cert khác.

📊 Visual — Revocation Flow
📝 Note & Conclusion

Certificate-based auth vượt trội hơn API key cho IoT và B2B: (1) Private key không bao giờ truyền qua network, (2) Revocation granular từng device, (3) Certificate có thể encode metadata (device model, location, tier) trong Subject/Extensions. OCSP Stapling cải thiện performance — server tự cache OCSP response, client không cần round-trip riêng để check revocation. Lưu ý: OCSP có thể là single point of failure — cần OCSP responder HA. CRL với TTL ngắn (1h) là fallback tốt.

02
Zero Trust — Không tin tưởng mặc định
"Never trust, always verify" — xác minh mọi request bất kể nguồn gốc
FRAMEWORK
💡 Định nghĩa
Zero Trust Security Model

Zero Trust là mô hình bảo mật dựa trên nguyên tắc: không có gì được tin tưởng mặc định — dù là user nội bộ, thiết bị trong LAN, hay service trong cùng datacenter. Mọi request đều phải được xác thực danh tính, kiểm tra quyền truy cập, và verify context (thiết bị có compliant không, vị trí địa lý, thời điểm) trước khi được phép.

Zero Trust ra đời từ nhận thức rằng perimeter-based security (firewall bảo vệ nội bộ) đã lỗi thời: với cloud, remote work, và insider threats, khái niệm "inside = trusted" không còn ý nghĩa.

7 Trụ cột của Zero Trust (NIST SP 800-207)

1. Identity

Mọi user, service, device phải có danh tính mạnh (strong auth: MFA, certificate, FIDO2). Identity là perimeter mới.

2. Device

Thiết bị phải đăng ký, compliant với policy (EDR, patch level, disk encryption). Chỉ managed devices được cấp quyền nhất định.

3. Network

Micro-segmentation thay vì flat network. Service-to-service communication kiểm soát bằng policy (mTLS + RBAC), không phải VLAN/firewall đơn thuần.

4. Workload

Application, container, serverless function đều có workload identity (SPIFFE). Không tin tưởng vì "chạy trong internal network".

5. Data

Classify và protect data (DLP, encryption at rest/transit). Access decisions dựa trên data classification, không chỉ user role.

6. Visibility

Log và monitor mọi access event. Anomaly detection, SIEM integration. Zero Trust không có giá trị nếu không có observability.

7. Automation

Policy enforcement tự động. Khi phát hiện anomaly, tự động revoke session, quarantine device, escalate. Con người không thể response đủ nhanh.

Case 2.1 — Zero Trust cho Remote Work: thay thế VPN truyền thốngINTERMEDIATE
📌 Problem Statement

Công ty có 2000 nhân viên làm remote. VPN truyền thống đặt ra vấn đề: (1) Sau khi kết nối VPN, nhân viên có access vào toàn bộ internal network — rủi ro nếu laptop bị compromise; (2) VPN không phân biệt BYOD vs managed device; (3) Khi bị hack, attacker có thể lateral move tự do trong mạng nội bộ; (4) VPN scaling khi 2000 người cùng connect.

✅ Solution — Zero Trust Network Access (ZTNA) thay thế VPN

Triển khai ZTNA (Cloudflare Access, Google BeyondCorp, Zscaler ZPA). Mỗi lần nhân viên truy cập ứng dụng: (1) Xác thực danh tính qua IdP (Okta/Azure AD) + MFA; (2) Đánh giá device posture (MDM compliance, OS patch level, EDR status); (3) Cấp quyền chỉ cho ứng dụng cụ thể được request, không phải toàn bộ network. Application connector chạy trong datacenter, không expose port ra internet.

📊 Visual — ZTNA vs VPN Access Model
📝 Note & Conclusion

ZTNA là evolution tự nhiên từ VPN. Điểm khác biệt cốt lõi: VPN cấp network access, ZTNA cấp application access. Với ZTNA, ngay cả khi laptop bị compromise, attacker chỉ có thể access những app user đó có quyền — không thể lateral move. Device posture check là component quan trọng: unmanaged/BYOD chỉ được access low-sensitivity apps. Thách thức: legacy apps không support modern auth cần ZTNA connector wrap. Migration từ VPN sang ZTNA nên làm dần dần, app-by-app.

RBAC — Role-Based Access Control
Phân quyền theo vai trò: đơn giản, dễ audit, dễ scale
AUTHORIZATION
💡 Định nghĩa
RBAC là gì?

Role-Based Access Control (RBAC) là mô hình kiểm soát truy cập trong đó quyền truy cập (permissions) được gán cho roles, và users được gán các roles phù hợp. User có quyền gì phụ thuộc vào role họ có, không phải được cấp trực tiếp. Đây là mô hình phổ biến nhất trong enterprise vì dễ quản lý và audit.

RBAC tuân theo nguyên tắc Least Privilege: user chỉ có quyền tối thiểu cần thiết cho công việc. Role tách biệt khỏi user — khi cần thay đổi quyền của một nhóm user, chỉ cần sửa role một lần.

Case 3.1 — RBAC cho SaaS Multi-tenant: mỗi tenant có role hierarchy riêngADVANCED
📌 Problem Statement

Một SaaS HR platform phục vụ 500 công ty (tenants). Mỗi công ty có cấu trúc tổ chức riêng và cần phân quyền khác nhau: HR Manager thấy lương của tất cả nhân viên trong phòng ban của mình nhưng không thấy phòng ban khác; CEO thấy tất cả; Employee chỉ thấy thông tin của bản thân. Cần thiết kế RBAC system không để tenant này access data của tenant khác (isolation) và cho phép mỗi tenant customize roles theo nhu cầu.

✅ Solution — Tenant-scoped RBAC với Resource-level Permissions

Mô hình: mỗi tenant có scope RBAC riêng. Role không phải global mà là tenant-specific. Permission bao gồm cả resource scope (chỉ department của mình). Token JWT chứa tenant_id + roles. Mọi query tự động filter theo tenant_id từ token — không cần developer nhớ thêm filter.

📊 Visual — Multi-tenant RBAC Model
📝 Note & Conclusion

Row-level tenant isolation phải được enforce ở tầng middleware/ORM, không chỉ ở application logic. Dùng PostgreSQL RLS (Row-Level Security) với current_setting('app.tenant_id') từ JWT claim để database tự động filter — không thể bị bypass ngay cả khi developer quên. Role explosion anti-pattern: nếu số lượng roles tăng theo cấp số nhân, cần xem lại thiết kế — thường là dấu hiệu cần chuyển sang ABAC cho phần fine-grained authorization.

04
ABAC — Attribute-Based Access Control
Policy engine đánh giá nhiều attribute — linh hoạt nhất
AUTHORIZATION
💡 Định nghĩa
ABAC là gì và tại sao vượt trội hơn RBAC trong nhiều tình huống?

Attribute-Based Access Control (ABAC) đưa ra quyết định phân quyền dựa trên tập hợp các thuộc tính (attributes) của Subject (người dùng), Resource (tài nguyên), Action (hành động), và Environment (context). Thay vì hỏi "User có role X không?", ABAC hỏi "User này, với các attributes này, trong context này, có được phép thực hiện action này trên resource này không?"

Ví dụ ABAC policy: "Doctor có thể xem medical record của patient nếu patient đang điều trị tại cùng hospital và doctor đang trong ca trực và access từ hospital network." — điều này không thể biểu diễn gọn gàng với RBAC.

4 loại Attributes trong ABAC

👤 Subject Attributes

Thuộc tính của người thực hiện request:

department: engineeringlevel: seniorclearance: secretlocation: hanoion_call: true
📄 Resource Attributes

Thuộc tính của tài nguyên được access:

classification: confidentialowner: alicedepartment: financesensitivity: high
⚡ Action Attributes

Thuộc tính của action được thực hiện:

type: readtype: writetype: deletetype: export
🌍 Environment Attributes

Context của request:

time: 09:00-18:00network: vpndevice: unmanagedrisk_score: low

Case 4.1 — ABAC cho Healthcare: bác sĩ chỉ xem hồ sơ bệnh nhân của mìnhEXPERT
📌 Problem Statement

Bệnh viện có 500 bác sĩ, 10,000 bệnh nhân. Yêu cầu phức tạp: Bác sĩ A có thể xem hồ sơ bệnh nhân X nếu X đang điều trị tại khoa của A VÀ A đang trong ca trực VÀ access từ thiết bị đã đăng ký. Bác sĩ cấp cứu có thể xem mọi hồ sơ nhưng chỉ trong 24h sau khi xử lý ca cấp cứu. RBAC không thể handle được độ phức tạp này.

✅ Solution — ABAC với OPA Policy kết hợp nhiều nguồn data

OPA policy đánh giá: (1) subject.department == resource.treating_department, (2) subject.on_duty == true (từ shift schedule system), (3) subject.device_registered == true (từ MDM), (4) hoặc subject.specialty == "emergency" AND time_since_treatment < 24h. Policy được test như unit test, deploy độc lập với application code.

📊 Visual — Decision Matrix
📝 Note & Conclusion

ABAC phù hợp nhất khi access control phụ thuộc vào nhiều yếu tố động mà RBAC không thể capture. Điểm mạnh: (1) Policy là code — test được, version control được; (2) Context-aware — xem xét time, location, device; (3) Centralized — một policy engine cho toàn bộ hệ thống. Điểm yếu: phức tạp hơn RBAC, cần đội ngũ hiểu Rego/policy language, OPA là external dependency cần HA. Practical recommendation: dùng RBAC cho coarse-grained authorization (user có access vào app không?), ABAC cho fine-grained (user có thể xem record cụ thể này không?).

OAuth 2.0 & OIDC
Ủy quyền và xác thực danh tính — tiêu chuẩn modern auth
AUTH PROTOCOL
💡 Phân biệt quan trọng
OAuth 2.0 ≠ OpenID Connect — Hai mục đích khác nhau

OAuth 2.0 là framework ủy quyền (authorization): cho phép application A được phép truy cập tài nguyên của user trên service B, không cần biết password của user. Câu hỏi: "Application này có được phép làm X thay mặt user không?"

OpenID Connect (OIDC) là identity layer xây dựng trên OAuth 2.0: thêm khả năng xác thực danh tính (authentication). Câu hỏi: "Ai là người đang login?". OIDC trả về ID Token (JWT chứa thông tin user), trong khi OAuth 2.0 trả về Access Token (quyền truy cập resource).

Case 5.1 — Client Credentials Flow: Service-to-Service auth không có userADVANCED
📌 Problem Statement

Hệ thống microservices có một notification-service cần gọi API của user-service để lấy email addresses vào lúc 2 AM — không có user nào đang login. Dùng API key đơn giản nhưng không có expiry, không có scope, khó rotate. Cần một cơ chế xác thực service-to-service đúng chuẩn với short-lived tokens và scope rõ ràng.

✅ Solution — OAuth 2.0 Client Credentials với scoped token

Mỗi service được đăng ký tại Authorization Server với client_id và client_secret. Khi cần gọi API, service tự request access token với scope cụ thể (users:read:email). Token có TTL ngắn (15 phút – 1 giờ). Resource server verify token và scope trước khi trả data. Tốt hơn API key vì: scope hạn chế, expiry ngắn, có thể revoke per-client.

📊 Visual — M2M Auth Flow
📝 Note & Conclusion

Client Credentials là pattern chuẩn cho M2M. Best practices: (1) client_secret lưu trong Secret Manager, không hardcode; (2) Cache token đến lúc gần expire (tránh request mỗi lần); (3) Scope hẹp nhất có thể — users:read:email thay vì users:*; (4) Token TTL ngắn (1h) với không có refresh token cho M2M — service phải request new token khi cần. Mutual TLS có thể thay thế hoàn toàn client_secret trong môi trường service mesh — mTLS + Client Credentials không dùng secret, chỉ dùng certificate.

JWT — JSON Web Token
Stateless token với signature — cấu trúc, verify, và anti-patterns nguy hiểm
TOKEN
💡 Định nghĩa
JWT là gì?

JSON Web Token (JWT) là chuẩn mở (RFC 7519) định nghĩa cách truyền thông tin giữa các party dưới dạng JSON object được ký số (signed). JWT có thể được verify mà không cần query database — đây là lợi thế lớn về scalability. Token gồm 3 phần: Header.Payload.Signature, được base64url-encode và nối với dấu chấm.

Cấu trúc JWT chi tiết

JWT Structure — Header . Payload . Signature

JWT Signing Algorithms — HS256 vs RS256 vs ES256

HS256 — HMAC SHA256

Symmetric: cùng một secret key để sign VÀ verify. Nhanh. Vấn đề: mọi service cần verify token phải biết secret key — nếu một service bị lộ, tất cả bị lộ. Chỉ dùng nếu chỉ có 1 service verify.

RS256 — RSA SHA256 (khuyến nghị)

Asymmetric: private key để sign (chỉ Auth Server có), public key để verify (publish qua JWKS endpoint). Service nào cũng verify được mà không biết private key. Tiêu chuẩn cho production.

ES256 — ECDSA SHA256

Asymmetric, signature nhỏ hơn RS256 (~50%), verify nhanh hơn. Tốt cho mobile/IoT nơi bandwidth và CPU quan trọng. Security tương đương RS256 với key nhỏ hơn nhiều.

🚨 JWT Anti-patterns Nguy Hiểm Nhất

⛔

Algorithm Confusion Attack (alg=none): Nếu server không kiểm tra alg trong header, attacker có thể đặt "alg":"none" và bỏ signature. Server accept token không có signature! Fix: Luôn whitelist algorithm cụ thể, reject nếu không match.

⚠️

Lưu JWT trong localStorage: Dễ bị XSS đánh cắp. Fix: Lưu access token trong memory (JavaScript variable), refresh token trong httpOnly cookie (không accessible từ JS). Đây là pattern được khuyến nghị nhất hiện tại.

Case 6.1 — Silent Token Refresh: access token ngắn + refresh token dàiADVANCED
📌 Problem Statement

Ứng dụng web dùng JWT access token 24 giờ. Nếu user bị hack, attacker có 24 giờ để dùng token — không thể revoke. Nếu đặt TTL 5 phút, user bị logout thường xuyên, trải nghiệm kém. Cần balance giữa security (token ngắn) và UX (không bị logout).

✅ Solution — Access Token 15 phút + Refresh Token 7 ngày (rotating)

Access token TTL ngắn (15 phút). Refresh token TTL dài (7 ngày), lưu trong httpOnly secure SameSite=strict cookie. Khi access token expire, client tự động dùng refresh token để lấy access token mới — silent refresh, user không bị gián đoạn. Rotation: mỗi lần dùng refresh token, issue token mới và invalidate token cũ — nếu token bị đánh cắp, sử dụng lần 2 sẽ bị detect.

📊 Visual — Token Refresh & Rotation Flow
📝 Note & Conclusion

Refresh token rotation là kỹ thuật quan trọng nhất để detect token theft: nếu attacker đánh cắp refresh token và dùng nó, server phát hiện "token đã được dùng rồi" → revoke toàn bộ token của user đó → attacker bị kick. Token Family: nhóm tất cả RT từ cùng login session vào một family — nếu bất kỳ RT nào trong family bị reuse, revoke toàn bộ family. httpOnly + SameSite=Strict cookie: không thể access từ JavaScript (chống XSS), không tự động gửi với cross-site request (chống CSRF).

Case 7.1 — Kubernetes + External Secrets Operator: sync từ AWS Secrets ManagerADVANCED
📌 Problem Statement

Team DevOps lưu secrets trong AWS Secrets Manager (compliance, audit, rotation tự động). Nhưng Kubernetes pods cần đọc secrets từ K8s Secrets object — nếu lưu song song ở cả hai nơi, dễ out-of-sync khi rotate. ✅ Solution — External Secrets Operator (ESO)

CORS — Cross-Origin Resource Sharing
Same-origin policy, preflight, headers, credentials
WEB SECURITY
💡 Định nghĩa
CORS là cơ chế browser, không phải server-side security

CORS (Cross-Origin Resource Sharing) là cơ chế browser cho phép hoặc từ chối web application từ một origin (https://app.com) gọi API tại origin khác (https://api.company.com). Browser mặc định chặn cross-origin requests theo Same-Origin Policy — CORS là cách server nới lỏng restriction này một cách có kiểm soát.

Quan trọng: CORS chỉ được enforce bởi browser. Curl, Postman, server-to-server calls không bị ảnh hưởng. CORS không phải là authentication hay authorization — nó chỉ kiểm soát browser nào được phép gọi API.

CORS — Use Cases & Pitfalls
Common misconfigurations, wildcard origins, credentials
USE CASES
Case 8.1 — CORS Misconfiguration: Wildcard + Credentials = Security HoleEXPERT
📌 Problem Statement

Developer cấu hình API server với Access-Control-Allow-Origin: * cho tiện, sau đó cũng thêm Access-Control-Allow-Credentials: true vì SPA cần gửi cookies. Ngạc nhiên: browser từ chối — báo lỗi "Wildcard origin cannot be used with credentials". Developer không hiểu tại sao và fix bằng cách chấp nhận mọi Origin header gửi lên: res.setHeader('Access-Control-Allow-Origin', req.headers.origin) — đây là lỗ hổng nghiêm trọng.

✅ Solution — Whitelist Origins + Validate + Vary Header

Duy trì danh sách allowed origins. Validate Origin header request so với whitelist. Chỉ phản hồi với Access-Control-Allow-Origin nếu origin trong whitelist. Luôn thêm Vary: Origin header để CDN cache đúng theo từng origin. Không bao giờ reflect Origin header mà không validate.

📊 Visual — CORS Security Matrix
📝 Note & Conclusion

Reflect Origin mà không validate = CORS-based CSRF vulnerability. Attacker có thể host evil.com, user visit → evil.com gọi api.company.com với user's cookies → read user data. Luôn validate Origin với whitelist cứng. Vary: Origin header bắt buộc khi respond khác nhau theo Origin — nếu không CDN cache response của origin A và trả cho origin B. CORS không phải authentication — kể cả cấu hình CORS đúng, vẫn cần authenticate request (JWT/session). CORS chỉ là browser enforcement, không bảo vệ direct API calls từ non-browser clients.

Case 8.2 — CORS cho Microservices: API Gateway là single CORS handlerINTERMEDIATE
📌 Problem Statement

Hệ thống có 15 microservices, mỗi service phải configure CORS riêng — dễ inconsistency, dễ misconfigure ở từng service. Team developer của các service không phải security expert. Cần centralized CORS policy, không duplicate ở mỗi service.

✅ Solution — Xử lý CORS tập trung tại API Gateway

API Gateway (Nginx, Kong, AWS API Gateway) xử lý tất cả CORS headers. Các internal microservices không cần biết về CORS — họ chỉ nhận requests đã qua gateway. Gateway intercept OPTIONS preflight, respond với CORS headers, forward actual requests sau khi validate origin. Internal services nhận request không có CORS concerns vì gateway đã handle.

📊 Visual — Centralized CORS at Gateway
📝 Note & Conclusion

API Gateway là điểm lý tưởng để xử lý CORS vì: (1) Single policy cho tất cả services; (2) Developer không cần hiểu CORS để viết services; (3) Dễ audit và thay đổi policy. Lưu ý với Load Balancer / CDN: nếu CDN cache preflight response, cần đảm bảo CDN không cache response sai origin (dùng Vary: Origin). SPA với multiple API domains: mỗi API domain cần CORS config riêng — không thể dùng wildcard nếu cần credentials. Xem xét BFF (Backend for Frontend) pattern để consolidate API calls qua một origin.
