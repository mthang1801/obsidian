Idempotency
Đảm bảo tính nhất quán khi retry — nền tảng của distributed systems
FOUNDATION
💡 Định nghĩa
Idempotency là gì?

Idempotency (tính lũy đẳng) là đặc tính của một operation mà khi thực hiện nhiều lần với cùng input, kết quả không thay đổi so với lần thực hiện đầu tiên. Trong toán học: f(f(x)) = f(x). Trong API: gọi cùng một request 1 lần hay 10 lần đều cho cùng kết quả.

Idempotency là nền tảng cốt lõi của mọi distributed system vì network không đáng tin cậy — packets có thể bị mất, duplicate, hoặc delay. Nếu không có idempotency, việc retry sẽ tạo ra side effects không mong muốn (ví dụ: tính tiền 2 lần).

Glossary chuyên sâu

Idempotency Key
Một unique identifier (UUID) gắn kèm mỗi request, cho phép server nhận diện request đã xử lý hay chưa. Client tạo key, server lưu trữ.
At-Least-Once Delivery
Message được đảm bảo gửi ít nhất 1 lần, nhưng có thể duplicate. Yêu cầu consumer phải idempotent.
At-Most-Once Delivery
Message chỉ gửi tối đa 1 lần, có thể bị mất. Không cần idempotency nhưng chấp nhận data loss.
Exactly-Once Semantics
Mỗi message được xử lý đúng 1 lần. Đạt được bằng cách kết hợp at-least-once + idempotent consumer.
Deduplication
Quá trình loại bỏ các request/message trùng lặp dựa trên idempotency key hoặc message ID.
Idempotent Consumer
Pattern trong messaging — consumer có khả năng xử lý cùng một message nhiều lần mà không tạo side effect.

Cách hoạt động

Idempotency Key Flow — Payment API
USE CASE CƠ BẢN
1.1 — Payment API với Idempotency Key
🔴 Problem Statement

Một fintech app cho phép user chuyển tiền. User bấm nút "Chuyển $100" nhưng network timeout — app không biết server đã xử lý hay chưa nên tự động retry. Không có idempotency → user bị trừ $200 (charge 2 lần). Đây là lỗi double-charge, vi phạm nghiêm trọng trong financial systems.

🟢 Solution

Client tạo một Idempotency-Key (UUID v4) trước khi gửi request. Server lưu key + response vào DB (với TTL 24h). Khi nhận request trùng key → trả cached response, không thực thi lại business logic.

Flow: Client → tạo Idempotency-Key: 550e8400-e29b... → gửi POST /transfers → Server check DB → nếu key chưa có: xử lý + lưu key/response → nếu key đã có: trả cached response.

📝 Note & Conclusion

Khi nào dùng: Mọi API có side effect (POST, PUT, DELETE) — đặc biệt payment, order creation, inventory deduction. Không cần cho GET vì GET đã tự nhiên idempotent.

Bài học: Idempotency key phải do client tạo (không phải server) để client có thể retry với cùng key. Key nên có TTL để tránh DB phình to.

USE CASE NÂNG CAO
1.2 — Idempotent Event Consumer trong Microservices
🔴 Problem Statement

Trong kiến trúc event-driven, Order Service publish event OrderCreated vào Kafka. Inventory Service consume event này để trừ stock. Kafka đảm bảo at-least-once delivery — nếu consumer crash trước khi commit offset, message sẽ được re-deliver → stock bị trừ 2 lần.

🟢 Solution

Áp dụng Idempotent Consumer Pattern: mỗi event có event_id duy nhất. Consumer duy trì bảng processed_events. Trước khi xử lý, check event_id: nếu đã xử lý → skip. Xử lý xong → INSERT event_id + COMMIT offset trong cùng một database transaction (atomic).

📝 Note & Conclusion

Critical: Check + process + save phải nằm trong cùng một transaction. Nếu tách ra, vẫn có race condition khi 2 consumer instances xử lý cùng event đồng thời. Dùng INSERT ... ON CONFLICT DO NOTHING hoặc UPSERT.

USE CASE EXPERT
1.3 — Distributed Idempotency với Redis + Lua Script
🔴 Problem Statement

Hệ thống e-commerce xử lý 50,000 orders/giây trên 20 server instances. Dùng PostgreSQL lưu idempotency keys → DB trở thành bottleneck vì mỗi request đều cần query + write. Cần giải pháp có latency < 1ms cho idempotency check.

🟢 Solution

Dùng Redis + Lua Script để đảm bảo atomic check-and-set: Lua script chạy SETNX key value EX ttl — nếu key chưa tồn tại → set + return "NEW"; nếu đã tồn tại → return "DUPLICATE". Toàn bộ logic chạy atomic trên Redis (single-threaded), không cần distributed lock. Kết hợp Redis cluster cho high availability.

📝 Note & Conclusion

Trade-off: Redis là in-memory → TTL phải hợp lý (24-48h cho payment, 5-10 min cho non-critical). Nếu Redis restart mà chưa persist → có thể mất key. Giải pháp: Redis AOF persistence + application-level fallback check vào DB.

Best Practices

  • HTTP Methods: GET, PUT, DELETE nên được thiết kế idempotent. POST cần idempotency key.
  • Idempotency key nên là UUID v4 hoặc UUID v7 (time-sortable) do client generate.
  • Server lưu key với TTL (24h cho payment, 1h cho non-critical) để tránh storage leak.
  • Response của idempotent request phải trả cùng HTTP status code và body — kể cả error response.
  • Dùng 409 Conflict nếu client gửi cùng key nhưng khác payload (request tampering).
  • Trong database: dùng UNIQUE constraint trên idempotency key column thay vì check-then-insert (race condition).
02
Eventual Consistency
Mô hình nhất quán cuối cùng trong hệ thống phân tán
FOUNDATION
💡 Định nghĩa
Eventual Consistency là gì?

Eventual Consistency là mô hình nhất quán dữ liệu trong đó hệ thống không đảm bảo tất cả các nodes sẽ có cùng dữ liệu tại mọi thời điểm, nhưng đảm bảo rằng nếu không có write mới, tất cả các nodes cuối cùng sẽ hội tụ về cùng một trạng thái (convergence).

Đây là trade-off cốt lõi trong CAP Theorem: để đạt high availability và partition tolerance, hệ thống chấp nhận dữ liệu có thể tạm thời không nhất quán giữa các replicas.

Glossary chuyên sâu

Strong Consistency
Mọi read sau write luôn trả về giá trị mới nhất. Đạt được bằng synchronous replication, nhưng tốn latency.
Causal Consistency
Các operations có quan hệ nhân quả (causal) được đảm bảo thứ tự. Mạnh hơn eventual, yếu hơn strong.
Read-Your-Writes
User luôn thấy dữ liệu mà chính mình đã write, dù hệ thống eventual consistent. Đạt bằng sticky sessions hoặc read-from-primary.
Convergence
Trạng thái cuối cùng khi tất cả replicas đồng bộ xong và có cùng dữ liệu. Thời gian convergence là "consistency window".
Consistency Window
Khoảng thời gian từ lúc write đến lúc tất cả replicas đồng bộ. Có thể từ milliseconds đến seconds tùy hệ thống.
Anti-Entropy
Cơ chế background synchronization giữa các replicas để phát hiện và sửa chữa data inconsistency (ví dụ: Merkle trees).

Strong vs Eventual Consistency

Strong Consistency

Ưu điểm: Dữ liệu luôn chính xác, dễ reasoning. Nhược điểm: High latency, lower availability, không scale tốt. Ví dụ: Banking transactions, inventory count, seat booking.

Eventual Consistency

Ưu điểm: Low latency, high availability, scale horizontally. Nhược điểm: Stale reads, conflict resolution phức tạp. Ví dụ: Social media feed, DNS, shopping cart, product catalog.

Eventual Consistency — Replication Timeline
USE CASE CƠ BẢN
2.1 — Social Media Feed: Like Count
🔴 Problem Statement

Facebook hiển thị like count trên mỗi post. Với hàng tỷ users, nếu mỗi like đều phải strong consistent → hệ thống phải lock row, synchronous replicate ra tất cả data centers → latency tăng cực kỳ cao, user experience tệ.

🟢 Solution

Dùng Eventual Consistency: khi user like, write vào local data center → async replicate ra các DC khác. User A ở Việt Nam thấy "1,000 likes", user B ở Mỹ có thể thấy "999 likes" trong vài giây. Không ai bị ảnh hưởng nghiêm trọng bởi sự chênh lệch này.

📝 Note & Conclusion

Nguyên tắc: Eventual consistency phù hợp khi "stale data không gây hại". Like count sai 1-2 trong vài giây không ảnh hưởng business. Nhưng account balance sai 1 đồng là thảm họa → cần strong consistency.

USE CASE NÂNG CAO
2.2 — E-commerce: Product Catalog + Inventory
🔴 Problem Statement

Shopee có catalog 500M sản phẩm, replicate qua 7 data centers. Nếu seller cập nhật giá sản phẩm, tất cả 7 DCs phải đồng bộ ngay → write latency quá cao (>500ms). Nhưng nếu eventual consistent cho inventory count, có thể bán oversell.

🟢 Solution

Hybrid approach: Product catalog (tên, mô tả, giá) dùng eventual consistency — stale vài giây chấp nhận được. Inventory count dùng strong consistency tại thời điểm checkout: khi user click "Mua", hệ thống read-from-primary để check stock chính xác, rồi dùng SELECT FOR UPDATE để lock row.

📝 Note & Conclusion

Bài học: Không phải toàn bộ hệ thống cần cùng một consistency level. Thiết kế theo per-field consistency: mỗi field có consistency requirement riêng dựa trên business impact.

USE CASE EXPERT
2.3 — CRDTs: Conflict-Free Replicated Data Types
🔴 Problem Statement

Trong collaborative editing (Google Docs), nhiều users edit cùng document đồng thời ở các regions khác nhau. Nếu dùng strong consistency → high latency, bad UX. Nếu dùng eventual consistency → write conflicts khi 2 users edit cùng vị trí.

🟢 Solution

Dùng CRDTs — data structures được thiết kế toán học để tự động merge mà không conflict. Ví dụ: G-Counter (grow-only counter) — mỗi node giữ counter riêng, tổng = sum tất cả nodes. OR-Set — add/remove tự động resolve bằng timestamps. CRDTs đảm bảo strong eventual consistency: mọi replicas nhận cùng set updates sẽ converge về cùng trạng thái, không cần consensus protocol.

📝 Note & Conclusion

Trade-off: CRDTs có overhead về storage (mỗi node giữ metadata). Chỉ phù hợp cho các data types đơn giản (counters, sets, registers). Cho complex business logic, vẫn cần Saga pattern hoặc consensus.

Best Practices

  • Phân loại data theo consistency requirement: critical (strong) vs non-critical (eventual).
  • Implement Read-Your-Writes cho UX tốt hơn — user luôn thấy data mình vừa write.
  • Dùng version vectors hoặc vector clocks để detect và resolve conflicts.
  • Monitor replication lag — nếu consistency window quá lớn (>30s), cần investigation.
  • Communicate rõ cho frontend team: data nào có thể stale, để hiển thị UI phù hợp (ví dụ: "~1,000 likes").

CAP Theorem
Consistency, Availability, Partition Tolerance — chọn 2 trong 3
FOUNDATION
💡 Định nghĩa
CAP Theorem là gì?

CAP Theorem (Brewer's Theorem, 2000) phát biểu rằng trong một distributed system, bạn chỉ có thể đảm bảo tối đa 2 trong 3 thuộc tính: Consistency (mọi node trả cùng data), Availability (mọi request đều được response), Partition Tolerance (hệ thống vẫn hoạt động khi network bị chia cắt).

Trong thực tế, Partition Tolerance là bắt buộc (network failures luôn xảy ra) → bạn thực sự chỉ chọn giữa CP (Consistency + Partition Tolerance) hoặc AP (Availability + Partition Tolerance).

Glossary chuyên sâu

Consistency (C)
Mọi read nhận được data mới nhất hoặc error. Tất cả nodes thấy cùng data tại cùng thời điểm (linearizability).
Availability (A)
Mọi request đến non-failing node đều nhận được response (có thể stale). Không có request nào bị từ chối.
Partition Tolerance (P)
Hệ thống tiếp tục hoạt động dù network bị split thành 2+ phần không thể giao tiếp với nhau.
PACELC Theorem
Mở rộng CAP: nếu có Partition → chọn A hoặc C; Else (bình thường) → chọn Latency hoặc Consistency. Thực tế hơn CAP.
Linearizability
Mức consistency mạnh nhất: mọi operation dường như xảy ra tại một thời điểm duy nhất, giữa invocation và response.
Quorum
Số nodes tối thiểu phải đồng ý cho một operation (W + R > N). Ví dụ: 3 nodes, quorum = 2. Dùng để tune CP vs AP.
CAP Theorem — Trade-off Triangle
USE CASE CƠ BẢN
3.1 — Banking System: Chọn CP
🔴 Problem Statement

Ngân hàng cần hệ thống transfer tiền giữa accounts. Nếu chọn AP (availability over consistency), khi network partition xảy ra, 2 partitions có thể approve cùng transfer → account balance âm, mất tiền.

🟢 Solution

Chọn CP system: khi partition xảy ra, hệ thống từ chối writes (return error) thay vì cho phép inconsistent writes. Dùng consensus protocol (Raft/Paxos) — chỉ partition có majority nodes mới được phục vụ. Minority partition trả 503 Service Unavailable. User bị ảnh hưởng tạm thời nhưng dữ liệu không bao giờ sai.

📝 Note & Conclusion

Quy tắc: Khi sai data tốn tiền nhiều hơn downtime → chọn CP. Banking, payment, inventory (physical goods) luôn là CP.

USE CASE NÂNG CAO
3.2 — DNS System: Chọn AP
🔴 Problem Statement

DNS phải phục vụ hàng tỷ queries/ngày. Nếu chọn CP, khi một root nameserver bị partition → toàn bộ internet segment không resolve được domain → thảm họa.

🟢 Solution

DNS chọn AP: mọi nameserver luôn respond (dù data có thể stale). TTL mechanism cho phép cached data expired sau thời gian nhất định. Khi partition heal, data tự convergence qua zone transfers. User có thể nhận stale IP trong vài phút nhưng service không bao giờ unavailable.

📝 Note & Conclusion

Quy tắc: Khi downtime tốn tiền nhiều hơn stale data → chọn AP. DNS, CDN, social feeds, product catalogs thường là AP.

USE CASE EXPERT
3.3 — PACELC: Vượt qua giới hạn CAP
🔴 Problem Statement

CAP chỉ nói về trường hợp partition. Nhưng 99.99% thời gian hệ thống hoạt động bình thường (no partition). Trong trường hợp bình thường, bạn vẫn phải trade-off giữa Latency vs Consistency.

🟢 Solution

PACELC Theorem: if Partition → choose A or C; Else → choose Latency or Consistency. Ví dụ: DynamoDB là PA/EL (partition → available; else → low latency). MongoDB là PC/EC (partition → consistent; else → consistent). Cassandra cho phép tune per-query: QUORUM reads = PC/EC, ONE reads = PA/EL.

📝 Note & Conclusion

Key insight: CAP là oversimplification. PACELC cho phép reasoning chính xác hơn. Khi thiết kế system, hãy xác định: (1) Khi partition: ưu tiên A hay C? (2) Bình thường: ưu tiên L hay C? Mỗi service/query có thể có câu trả lời khác nhau.

Best Practices

  • Không có "CA distributed system" thực tế — network partition luôn có thể xảy ra.
  • Thiết kế per-service CAP choice: Payment → CP, Product Catalog → AP, User Profile → AP.
  • Dùng Quorum reads/writes để fine-tune giữa CP và AP trong cùng một database.
  • Monitor partition events và có runbook cho mỗi scenario (split-brain, network flap).
  • Trong interviews, luôn mention PACELC thay vì chỉ CAP — thể hiện hiểu biết sâu hơn.
04
Saga Pattern
Quản lý distributed transactions qua chuỗi compensating actions
ADVANCED
💡 Định nghĩa
Saga Pattern là gì?

Saga là pattern quản lý distributed transactions bằng cách chia một transaction lớn thành chuỗi local transactions nhỏ. Mỗi local transaction update một service và publish event/message để trigger transaction tiếp theo. Nếu một step thất bại, saga thực hiện compensating transactions (rollback ngược) cho các steps đã hoàn thành.

Saga thay thế 2-Phase Commit (2PC) — vốn không khả thi trong microservices vì tạo tight coupling và single point of failure.

Glossary chuyên sâu

Choreography Saga
Mỗi service tự lắng nghe events và quyết định bước tiếp theo. Không có central coordinator. Phù hợp saga đơn giản (2-4 steps).
Orchestration Saga
Một Saga Orchestrator trung tâm điều phối toàn bộ flow, gửi commands đến từng service. Phù hợp saga phức tạp (5+ steps).
Compensating Transaction
Operation ngược lại để "undo" một local transaction đã commit. Ví dụ: Debit → Credit back. Phải idempotent.
Semantic Rollback
Không phải DB rollback thực sự (đã commit rồi), mà là business-level undo. Ví dụ: cancel order, refund payment.
Pivot Transaction
Transaction quyết định saga thành công hay thất bại. Trước pivot: có thể compensate. Sau pivot: chỉ có retryable transactions.
Saga Execution Coordinator (SEC)
Component lưu trữ saga state, quyết định step tiếp theo, xử lý failures. Thường dùng state machine.
Orchestration Saga — Order Processing Flow
USE CASE CƠ BẢN
4.1 — E-commerce Order: Choreography Saga
🔴 Problem Statement

Hệ thống e-commerce có 3 services: Order, Payment, Inventory. Khi user đặt hàng, cần: (1) tạo order, (2) charge tiền, (3) trừ inventory. Nếu inventory hết hàng sau khi đã charge → phải refund. Không thể dùng distributed transaction (2PC) vì mỗi service có DB riêng.

🟢 Solution

Choreography Saga: Order Service tạo order + publish OrderCreated event → Payment Service listen, charge tiền + publish PaymentCharged → Inventory Service listen, reserve stock. Nếu Inventory fail → publish StockReservationFailed → Payment Service listen, refund → Order Service listen, cancel order. Mỗi service tự biết phải làm gì khi nhận event.

📝 Note & Conclusion

Ưu điểm: Loose coupling, simple. Nhược điểm: Khó debug vì logic phân tán. Khi saga có 5+ steps → spaghetti events, khó hiểu flow. → Chuyển sang Orchestration.

USE CASE NÂNG CAO
4.2 — Travel Booking: Orchestration Saga
🔴 Problem Statement

Booking platform cần đặt: Flight + Hotel + Car rental. Mỗi service là third-party API (Amadeus, Booking.com, Hertz). Nếu flight booked nhưng hotel full → phải cancel flight. Flow phức tạp với 6+ steps, nhiều edge cases.

🟢 Solution

Orchestration Saga với state machine: Orchestrator giữ saga state (PENDING → FLIGHT_BOOKED → HOTEL_BOOKED → CAR_BOOKED → COMPLETED). Mỗi state transition gửi command đến service tương ứng. Nếu step fail, orchestrator biết chính xác đã hoàn thành steps nào → gửi compensating commands ngược lại. State được persist vào DB để survive crashes.

📝 Note & Conclusion

Key design: Mỗi saga step phải có compensating action tương ứng. Compensating actions phải idempotent (có thể retry). Orchestrator state phải persisted (not in-memory) để survive restarts.

USE CASE EXPERT
4.3 — Saga State Machine với Dead Letter Queue
🔴 Problem Statement

Trong production, compensating transaction cũng có thể fail. Ví dụ: cần refund payment nhưng payment gateway đang down. Saga bị stuck ở trạng thái "partially compensated" — không thể hoàn thành, cũng không thể rollback.

🟢 Solution

Thiết kế retry + DLQ (Dead Letter Queue): compensating actions được retry với exponential backoff (1s → 2s → 4s → max 5 min). Sau N retries fail → message chuyển vào DLQ. Monitoring alert ops team. Manual intervention hoặc scheduled job xử lý DLQ. Saga state chuyển sang COMPENSATION_FAILED — cần human review. Bổ sung saga timeout: nếu saga không complete trong 24h → auto-escalate.

📝 Note & Conclusion

Reality check: Saga không phải silver bullet. Nó đưa complexity từ database layer lên application layer. Hãy chắc chắn bạn thực sự cần distributed transactions trước khi dùng Saga. Nhiều trường hợp, một database duy nhất với ACID transactions là giải pháp tốt nhất.

05
CQRS
Command Query Responsibility Segregation — tách biệt đọc và ghi
ADVANCED
💡 Định nghĩa
CQRS là gì?

CQRS (Command Query Responsibility Segregation) là pattern tách biệt write model (Command) và read model (Query) thành hai models riêng biệt, có thể sử dụng khác data stores, schemas, và scaling strategies. Commands thay đổi state, Queries đọc state — hai responsibilities này có requirements rất khác nhau.

Glossary chuyên sâu

Command
Request thay đổi state. Ví dụ: CreateOrder, UpdateProfile. Không trả về data (chỉ success/failure). Validated trước khi execute.
Query
Request đọc data. Ví dụ: GetOrderById, SearchProducts. Không thay đổi state. Có thể cached, replicated, denormalized.
Write Model
Domain model tối ưu cho business rules, validation, invariants. Normalized, ACID. Thường dùng RDBMS.
Read Model
View model tối ưu cho queries. Denormalized, pre-computed. Có thể dùng Elasticsearch, Redis, materialized views.
Projection
Process biến đổi write events thành read model. Ví dụ: OrderCreated event → update order_summary_view table.
Thin Read Layer
Read side không có business logic, chỉ query denormalized data. Rất nhanh, dễ scale, dễ cache.
CQRS Architecture — Separated Read/Write Paths
USE CASE CƠ BẢN
5.1 — E-commerce: Product Listing với Read-Optimized Store
🔴 Problem Statement

Product page cần hiển thị: tên, giá, rating, review count, seller info, shipping options — dữ liệu nằm ở 6 tables khác nhau. Mỗi page view = 6 JOINs. Với 100K concurrent users → database không chịu nổi.

🟢 Solution

Tách thành Write DB (PostgreSQL, normalized) cho product management + Read DB (Elasticsearch) cho product search/listing. Khi seller update product → write vào PostgreSQL → CDC/event publish → Projection service update Elasticsearch document (denormalized, chứa tất cả data cần hiển thị). Query chỉ cần 1 request vào ES — no JOINs, sub-millisecond latency.

📝 Note & Conclusion

Trade-off: Read model sẽ eventually consistent so với write model. Seller update giá → user có thể thấy giá cũ trong 1-2 giây. Chấp nhận được cho product catalog, không chấp nhận cho inventory count.

USE CASE EXPERT
5.2 — CQRS + Event Sourcing: Full Audit Trail
🔴 Problem Statement

Hệ thống tài chính cần: (1) full audit trail mọi thay đổi, (2) khả năng reconstruct state tại bất kỳ thời điểm nào (point-in-time recovery), (3) multiple read views (dashboard, report, compliance). Traditional CRUD chỉ lưu state hiện tại, không có history.

🟢 Solution

CQRS + Event Sourcing: Write side lưu events (không lưu state) vào append-only event store. Mỗi event = fact đã xảy ra (AccountDebited $100, AccountCredited $50). Read side: nhiều projections khác nhau consume cùng event stream để tạo các views khác nhau — dashboard projection, report projection, compliance projection. Mỗi projection có thể rebuild từ đầu bằng cách replay tất cả events.

📝 Note & Conclusion

Warning: CQRS + Event Sourcing là một trong những patterns phức tạp nhất. Chỉ dùng khi business thực sự cần audit trail + multiple views + temporal queries. Cho CRUD apps đơn giản, đây là massive over-engineering.

Best Practices

  • Bắt đầu với simple CQRS: cùng DB nhưng tách read/write models ở application level.
  • Chỉ dùng separate databases khi read/write có scaling requirements thực sự khác nhau.
  • Read model projections phải idempotent — có thể rebuild bất cứ lúc nào.
  • Đảm bảo UI handle eventual consistency gracefully (optimistic updates, loading states).
  • CQRS không yêu cầu Event Sourcing — hai patterns này independent nhưng complementary.
06
Event Sourcing
Lưu trữ state dưới dạng chuỗi immutable events
ADVANCED
💡 Định nghĩa
Event Sourcing là gì?

Event Sourcing là pattern lưu trữ state của một entity không phải bằng current state (row trong DB), mà bằng chuỗi tất cả events (facts) đã xảy ra. State hiện tại được reconstruct bằng cách replay tất cả events từ đầu. Events là immutable — không bao giờ update hay delete.

Tương tự như git: bạn không lưu file cuối cùng, mà lưu tất cả commits. Muốn xem state tại bất kỳ thời điểm → checkout commit tương ứng.

Glossary chuyên sâu

Event Store
Database chuyên dụng lưu events. Append-only, ordered by sequence. Ví dụ: EventStoreDB, Kafka, PostgreSQL (append-only table).
Aggregate Stream
Chuỗi events thuộc về một aggregate instance. Ví dụ: tất cả events của Order #123 tạo thành một stream.
Snapshot
Periodic checkpoint lưu current state để tránh replay toàn bộ events. Ví dụ: snapshot mỗi 100 events.
Event Upcasting
Chuyển đổi event schema cũ sang schema mới khi event format thay đổi. Giống database migration cho events.
Temporal Query
Query state tại một thời điểm trong quá khứ bằng cách replay events đến thời điểm đó. Unique capability của event sourcing.
Event Replay
Quá trình tái tạo state bằng cách apply tất cả events tuần tự. Dùng để rebuild read models hoặc debug.
Event Sourcing — Account Balance Reconstruction
USE CASE CƠ BẢN
6.1 — Banking Ledger: Full Transaction History
🔴 Problem Statement

Ngân hàng cần lưu toàn bộ lịch sử giao dịch của mỗi tài khoản cho mục đích audit, compliance, và dispute resolution. Traditional CRUD chỉ lưu balance hiện tại — nếu có bug tính sai, không thể trace nguyên nhân.

🟢 Solution

Dùng Event Sourcing: mỗi transaction là một event (Deposited, Withdrawn, Transferred). Balance = replay tất cả events. Mọi thay đổi đều có trace. Khi phát hiện bug → replay events với fixed logic → tự động tính lại balance chính xác. Regulators có thể audit toàn bộ history.

📝 Note & Conclusion

Performance: Replay 1M events để tính balance rất chậm. Giải pháp: Snapshots — lưu balance tại mỗi 1000 events. Reconstruct = load snapshot + replay events sau snapshot.

USE CASE EXPERT
6.2 — Event Schema Evolution & Upcasting
🔴 Problem Statement

Hệ thống đã chạy 3 năm, có 500M events. Event OrderCreated v1 có field price (single currency). Business mở rộng multi-currency → cần price thành {amount, currency}. Không thể update 500M events (immutable). Code mới cần đọc cả events cũ và mới.

🟢 Solution

Event Upcasting: Khi đọc event cũ, upcast middleware tự động transform v1 → v2: {price: 100} → {amount: 100, currency: "USD"} (default currency). Events trong store vẫn giữ nguyên (immutable). Code chỉ cần handle event v2. Upcaster chain: v1→v2→v3 cho multiple migrations. Đây là "database migration" cho event sourcing.

📝 Note & Conclusion

Key challenge: Event schema evolution là khó nhất của event sourcing. Events là immutable contracts — thay đổi schema phải backward compatible. Dùng schema registry (Avro/Protobuf) để enforce compatibility.

07
Outbox Pattern
Đảm bảo reliable messaging giữa DB write và event publish
ADVANCED
💡 Định nghĩa
Outbox Pattern là gì?

Outbox Pattern giải quyết bài toán dual write: khi cần vừa write vào database vừa publish event vào message broker, hai operations này không nằm trong cùng một transaction → có thể một thành công, một thất bại → data inconsistency.

Giải pháp: thay vì publish trực tiếp vào broker, write event vào outbox table trong cùng DB transaction với business data. Một background process (poller hoặc CDC) đọc outbox table và publish vào broker.

Glossary chuyên sâu

Dual Write Problem
Khi cần write vào 2 systems (DB + broker) mà không có distributed transaction. Nếu một fail → inconsistency.
Outbox Table
Table trong cùng DB chứa pending events. Columns: id, aggregate_type, aggregate_id, event_type, payload, created_at, published_at.
Polling Publisher
Background job query outbox table định kỳ (mỗi 100ms-1s), publish unpublished events, mark as published. Simple nhưng có delay.
CDC (Change Data Capture)
Capture changes từ DB transaction log (WAL/binlog) real-time. Ví dụ: Debezium đọc PostgreSQL WAL → publish vào Kafka. No polling delay.
Transaction Log Tailing
Đọc database's internal transaction log để detect inserts vào outbox table. Real-time, no additional queries. Debezium, Maxwell, Canal.
Exactly-Once Publishing
Đảm bảo mỗi event chỉ publish một lần. Đạt bằng cách marking outbox row as published trong cùng transaction với ack.
Outbox Pattern — Transactional Messaging Flow
USE CASE CƠ BẢN
7.1 — Order Service: DB + Event Consistency
🔴 Problem Statement

Order Service cần: (1) INSERT order vào DB, (2) Publish OrderCreated event vào Kafka. Nếu INSERT thành công nhưng Kafka publish fail → order tồn tại nhưng không ai biết. Nếu publish trước rồi INSERT fail → event đã gửi nhưng order không tồn tại. Dual write = data inconsistency.

🟢 Solution

Dùng Outbox Pattern: trong cùng DB transaction, INSERT vào orders table VÀ INSERT vào outbox table. Background poller (mỗi 500ms) query outbox WHERE published_at IS NULL → publish vào Kafka → UPDATE published_at = NOW(). Nếu poller crash trước update → event chưa mark published → sẽ re-publish (at-least-once). Consumer phải idempotent.

📝 Note & Conclusion

Key insight: Outbox biến dual write thành single write (cùng DB). Reliability chuyển từ "DB + Broker" thành "chỉ DB" (ACID). Trade-off: thêm latency (polling interval) và thêm table trong DB.

USE CASE EXPERT
7.2 — Debezium CDC: Zero-Latency Outbox
🔴 Problem Statement

Polling outbox mỗi 500ms tạo delay. Với hệ thống xử lý 10K orders/giây, polling query trở thành bottleneck. Cần real-time event publishing mà vẫn đảm bảo transactional consistency.

🟢 Solution

Dùng Debezium CDC Connector: Debezium đọc PostgreSQL WAL (Write-Ahead Log) real-time, detect INSERT vào outbox table → tự động publish vào Kafka. Latency: milliseconds thay vì seconds. Không cần polling query → no DB load. Debezium giữ offset trong Kafka → crash-safe. Kết hợp Outbox Event Router SMT (Single Message Transform) để transform outbox row thành proper event format.

📝 Note & Conclusion

Production stack: PostgreSQL + Debezium + Kafka là gold standard cho transactional messaging. Trade-off: thêm infrastructure complexity (Debezium connector, Kafka Connect cluster). Cho small teams, polling publisher đủ tốt.

08
Strangler Fig Pattern
Migration từ monolith sang microservices một cách an toàn
ADVANCED
💡 Định nghĩa
Strangler Fig Pattern là gì?

Strangler Fig Pattern (đặt tên theo cây Strangler Fig — loài cây bắt đầu mọc bám trên cây chủ rồi dần thay thế hoàn toàn) là chiến lược migration incremental từ legacy system sang system mới. Thay vì rewrite toàn bộ (Big Bang), bạn dần dần chặn và chuyển hướng từng feature sang service mới cho đến khi legacy system không còn được sử dụng.

Glossary chuyên sâu

Strangler Facade
Layer proxy đứng trước cả legacy và new system, routing requests dựa trên feature flags hoặc URL paths.
Feature Toggle
Cơ chế bật/tắt routing giữa legacy và new system per-feature. Cho phép gradual rollout và instant rollback.
Anti-Corruption Layer (ACL)
Adapter layer giữa new service và legacy system. Translate giữa new domain model và legacy data format.
Shadow Traffic
Gửi copy của production traffic sang new system để test nhưng không trả response cho client. Dùng để validate correctness.
Strangler Fig — Phased Migration
USE CASE NÂNG CAO
8.1 — Monolith to Microservices: Gradual Migration
🔴 Problem Statement

Công ty có monolith PHP 8 năm tuổi, 2M LOC. Muốn chuyển sang Go microservices nhưng: (1) Big Bang rewrite mất 2 năm và risk cực cao, (2) Không thể freeze features 2 năm, (3) Team vẫn phải deliver business features trong lúc migrate.

🟢 Solution

Strangler Fig: Đặt API Gateway (NGINX/Kong) trước monolith. Migrate từng bounded context: bắt đầu với module ít coupling nhất (ví dụ: Notification). Route /api/notifications/* sang new Go service, phần còn lại vẫn vào monolith. Mỗi sprint migrate 1-2 modules. Dùng feature flags để A/B test new vs legacy. Sau 12 tháng, monolith chỉ còn lại core module → migrate cuối cùng → decommission.

📝 Note & Conclusion

Critical success factor: Chọn module đầu tiên phải là module ít dependency nhất. Dùng Anti-Corruption Layer để new service không bị "nhiễm" legacy data model. Database: ban đầu new service có thể đọc từ legacy DB (shared database), sau đó tách dần sang DB riêng.

09
Circuit Breaker
Bảo vệ hệ thống khỏi cascading failures
ADVANCED
💡 Định nghĩa
Circuit Breaker là gì?

Circuit Breaker là resilience pattern lấy cảm hứng từ cầu dao điện: khi phát hiện downstream service liên tục fail, circuit breaker "ngắt mạch" — tất cả requests tiếp theo bị reject ngay lập tức (fail-fast) thay vì chờ timeout. Sau một khoảng thời gian, circuit breaker cho phép một vài requests thử lại (half-open) để kiểm tra service đã recover chưa.

Glossary chuyên sâu

Closed State
Trạng thái bình thường. Requests đi qua. Circuit breaker đếm failures. Khi failure count vượt threshold → chuyển sang Open.
Open State
Trạng thái "ngắt mạch". Tất cả requests bị reject ngay (không gọi downstream). Sau timeout → chuyển sang Half-Open.
Half-Open State
Cho phép một số requests thử lại. Nếu thành công → Closed. Nếu fail → Open lại. Probe mechanism để test recovery.
Failure Threshold
Số failures liên tiếp hoặc failure rate (%) để trigger Open. Ví dụ: 5 failures liên tiếp hoặc 50% failure rate trong 60s.
Recovery Timeout
Thời gian ở Open state trước khi chuyển Half-Open. Thường 30-60 giây. Cho downstream time to recover.
Fallback
Response thay thế khi circuit Open. Ví dụ: cached data, default value, degraded response. Đảm bảo UX vẫn acceptable.
Circuit Breaker — State Machine
USE CASE CƠ BẢN
9.1 — Payment Gateway: Fail-Fast Protection
🔴 Problem Statement

Order Service gọi Payment Gateway (Stripe). Stripe đang bị outage — mỗi request timeout sau 30 giây. Với 1000 concurrent users, Order Service tạo 1000 threads chờ timeout → thread pool exhaustion → Order Service cũng chết → cascading failure lan sang toàn hệ thống.

🟢 Solution

Wrap Stripe calls trong Circuit Breaker: threshold = 5 consecutive failures, timeout = 30s. Sau 5 lần Stripe fail → circuit Open → tất cả payment requests tiếp theo nhận ngay error "Payment service temporarily unavailable" (fail trong 1ms thay vì 30s). Sau 30s → Half-Open → thử 1 request → nếu Stripe recovered → Closed → bình thường. Order Service sống sót, user nhận feedback nhanh.

📝 Note & Conclusion

Giá trị cốt lõi: Circuit Breaker biến slow failure thành fast failure. Slow failure (30s timeout) nguy hiểm hơn fast failure (1ms reject) rất nhiều vì nó giữ resources (threads, connections, memory).

USE CASE EXPERT
9.2 — Adaptive Circuit Breaker với Sliding Window
🔴 Problem Statement

Simple circuit breaker (count-based) có vấn đề: nếu 5 failures xảy ra cách nhau 1 giờ → circuit Open, dù failure rate thực tế rất thấp (5/10000 = 0.05%). Cần circuit breaker thông minh hơn dựa trên failure rate, không phải absolute count.

🟢 Solution

Sliding Window Circuit Breaker: theo dõi failure rate trong sliding window (ví dụ: 60 giây, minimum 20 calls). Chỉ trip khi failure rate > 50% VÀ minimum calls > 20. Kết hợp slow call rate: nếu >80% calls mất >5s → cũng trip (dù không error, nhưng downstream đang chậm). Libraries: resilience4j (Java), sony/gobreaker (Go), polly (.NET).

📝 Note & Conclusion

Best practice: Expose circuit breaker metrics (state, failure rate, call count) vào monitoring (Prometheus/Grafana). Alert khi circuit Opens. Log mọi state transitions. Đây là critical observability cho production systems.

10
Bulkhead Pattern
Cô lập failures bằng resource partitioning
ADVANCED
💡 Định nghĩa
Bulkhead Pattern là gì?

Bulkhead (vách ngăn tàu) là pattern cô lập các components của hệ thống vào resource pools riêng biệt, để khi một component fail, nó không consume hết resources của toàn hệ thống. Tương tự vách ngăn trên tàu: khi một khoang bị thủng, nước chỉ tràn vào khoang đó, không chìm toàn bộ tàu.

Glossary chuyên sâu

Thread Pool Bulkhead
Mỗi dependency có thread pool riêng. Payment calls: 20 threads. Inventory calls: 10 threads. Nếu Payment timeout → chỉ 20 threads bị block.
Semaphore Bulkhead
Dùng semaphore giới hạn concurrent calls. Lightweight hơn thread pool (no thread creation overhead). Chạy trên caller's thread.
Connection Pool Isolation
Mỗi service/database có connection pool riêng. DB query chậm chỉ exhaust pool của nó, không ảnh hưởng pools khác.
Process Isolation
Chạy components trong processes riêng (containers). Heaviest isolation: memory, CPU, network all separated.
Bulkhead — Resource Pool Isolation
USE CASE NÂNG CAO
10.1 — API Gateway: Per-Service Rate Limiting
🔴 Problem Statement

API Gateway forward requests đến 5 downstream services. Recommendation Service bị chậm (ML inference heavy) → gateway dùng hết connection pool chờ recommendations → các services khác cũng không được phục vụ dù chúng hoàn toàn healthy.

🟢 Solution

Connection Pool Bulkhead: Gateway tạo separate connection pool cho mỗi downstream service: Recommendation (20 connections), Product (50), User (30), Order (50), Search (30). Khi Recommendation pool đầy → chỉ recommendation requests bị reject (với fallback: hiển thị trending products thay vì personalized). Các services khác không bị ảnh hưởng vì dùng pools riêng.

📝 Note & Conclusion

Kết hợp với Circuit Breaker: Bulkhead (giới hạn concurrent calls) + Circuit Breaker (fast-fail khi downstream dead) = robust resilience. Bulkhead giới hạn blast radius, Circuit Breaker giới hạn duration of impact.

Sidecar Pattern
Tách cross-cutting concerns vào container phụ trợ
EXPERT
💡 Định nghĩa
Sidecar Pattern là gì?

Sidecar là pattern triển khai helper component song song cùng main application trong cùng một host (thường là cùng Pod trong Kubernetes). Sidecar xử lý cross-cutting concerns như logging, monitoring, networking, security — cho phép main app tập trung vào business logic. Tương tự xe máy có thùng phụ (sidecar) chở thêm hành lý.

Glossary chuyên sâu

Sidecar Proxy
Proxy chạy bên cạnh app, intercept tất cả inbound/outbound traffic. Ví dụ: Envoy proxy trong Istio. Xử lý mTLS, load balancing, retry.
Init Container
Container chạy trước main container để setup (ví dụ: inject iptables rules cho traffic redirection). Khác sidecar: chạy một lần rồi exit.
Ambassador Pattern
Variant của Sidecar — proxy outbound calls. Ví dụ: sidecar handle connection pooling, retry, circuit breaking cho calls đến external services.
Adapter Pattern
Variant của Sidecar — transform data format. Ví dụ: sidecar convert logs từ app-specific format sang standardized format cho central logging.
Sidecar Pattern — Kubernetes Pod Architecture
USE CASE NÂNG CAO
11.1 — Polyglot Observability với Sidecar
🔴 Problem Statement

Tổ chức có 30+ microservices viết bằng Go, Java, Python, Node.js. Mỗi team dùng logging library khác nhau, metrics format khác nhau. Cần unified observability nhưng không thể yêu cầu tất cả teams refactor code.

🟢 Solution

Observability Sidecar: Deploy sidecar container (OpenTelemetry Collector) trong mỗi pod. Sidecar thu thập: (1) Logs từ stdout/files → ship đến Loki/ELK, (2) Metrics từ /metrics endpoint → ship đến Prometheus, (3) Traces qua OTLP → ship đến Jaeger. App chỉ cần output logs ra stdout và expose metrics endpoint — sidecar handle tất cả formatting, batching, shipping. Zero code change cho existing services.

📝 Note & Conclusion

Trade-off: Mỗi pod thêm 1 container → tốn RAM (~30-100MB/pod) và CPU. Với 500 pods = thêm 15-50GB RAM cluster-wide. Nhưng đổi lại: unified observability, language-agnostic, no code changes. Giá trị >> cost.

12
Service Mesh
Infrastructure layer cho service-to-service communication
EXPERT
💡 Định nghĩa
Service Mesh là gì?

Service Mesh là dedicated infrastructure layer xử lý service-to-service communication, bao gồm: service discovery, load balancing, encryption (mTLS), observability, traffic management, và resilience (retry, circuit breaking). Nó triển khai bằng cách inject sidecar proxy vào mỗi service — tạo thành một "mesh" (lưới) proxy quản lý toàn bộ network traffic.

Glossary chuyên sâu

Data Plane
Tập hợp tất cả sidecar proxies (Envoy). Xử lý actual traffic: routing, load balancing, TLS, metrics. Nằm trong mỗi pod.
Control Plane
Centralized management layer (Istiod). Cấu hình policies, distribute certificates, push routing rules đến data plane.
mTLS (mutual TLS)
Cả client và server đều verify certificate. Service mesh auto-manage certificates: generation, distribution, rotation. Zero-trust networking.
Traffic Splitting
Route % traffic giữa versions. Ví dụ: 95% → v1, 5% → v2 (canary deployment). Managed tại mesh level, transparent to apps.
Istio
Service mesh phổ biến nhất, dùng Envoy proxy. Feature-rich nhưng complex. Phù hợp large-scale deployments (100+ services).
Linkerd
Lightweight service mesh, dùng Rust-based proxy (linkerd2-proxy). Simpler than Istio, lower resource overhead. CNCF graduated project.
Service Mesh Architecture — Data Plane + Control Plane
USE CASE EXPERT
12.1 — Zero-Trust Security với mTLS Mesh
🔴 Problem Statement

Tổ chức tài chính có 200+ microservices trong Kubernetes. Security team yêu cầu: (1) encrypt mọi traffic in-transit, (2) authenticate mọi service call (service-to-service), (3) RBAC: Service A chỉ được gọi Service B, không được gọi Service C. Implement vào mỗi service = enormous effort (200 services × multiple languages).

🟢 Solution

Istio Service Mesh: Deploy Istio → auto-inject Envoy sidecar vào tất cả pods. Istio CA auto-generate mTLS certificates cho mỗi service. Mọi traffic giữa services tự động encrypted + authenticated. Define AuthorizationPolicy: chỉ cho phép Order Service → Payment Service (deny all by default). Zero code changes. Certificates auto-rotated mỗi 24h.

📝 Note & Conclusion

Khi nào cần Service Mesh: 50+ services, polyglot (multiple languages), security requirements (mTLS mandatory), complex traffic routing (canary, A/B). Dưới 10 services → overkill, dùng library-based approach (ví dụ: gRPC interceptors).

13
API Gateway
Single entry point cho microservices — routing, auth, rate limiting
ADVANCED
💡 Định nghĩa
API Gateway là gì?

API Gateway là server đóng vai trò single entry point cho tất cả client requests. Nó nhận request từ clients, route đến appropriate microservice, aggregate responses nếu cần, và xử lý cross-cutting concerns: authentication, rate limiting, SSL termination, caching, request/response transformation, logging.

Thay vì client gọi trực tiếp 10 microservices (phải biết address, handle auth, retry cho mỗi service), client chỉ gọi 1 gateway. Gateway = "front door" của hệ thống.

Glossary chuyên sâu

Reverse Proxy
Server nhận requests từ clients và forward đến backend services. API Gateway là reverse proxy + thêm nhiều features.
Request Aggregation
Gateway gọi multiple services song song và merge responses thành 1 response trả cho client. Giảm number of round trips.
Rate Limiting
Giới hạn số requests per client/IP/API key trong time window. Ví dụ: 1000 req/minute. Bảo vệ backend khỏi overload.
SSL/TLS Termination
Gateway xử lý SSL handshake, decrypt HTTPS. Backend services communicate qua HTTP (internal network). Giảm TLS overhead cho backends.
API Composition
Pattern gateway gọi multiple services để compose một API response mà không service nào có đủ data. Ví dụ: product + reviews + seller info.
Kong / APISIX / Traefik
Popular open-source API Gateways. Kong (Lua/NGINX), APISIX (Lua/NGINX), Traefik (Go). Cloud: AWS API Gateway, Azure API Management.
API Gateway — Microservices Entry Point
USE CASE CƠ BẢN
13.1 — Centralized Authentication
🔴 Problem Statement

10 microservices, mỗi service tự implement JWT validation. Khi cần thay đổi auth logic (ví dụ: thêm API key support, rotate signing keys) → phải update 10 services, deploy 10 lần. Inconsistent auth → security gaps.

🟢 Solution

API Gateway handles authentication: Gateway validate JWT token, extract user info, attach vào request headers (X-User-Id, X-User-Role). Backend services trust headers từ gateway (internal network). Thay đổi auth logic: chỉ cần update gateway. Backend services đơn giản hóa: không cần biết JWT, chỉ đọc headers.

📝 Note & Conclusion

Security: Backend services PHẢI chỉ accept requests từ gateway (network policy). Nếu attacker bypass gateway gọi trực tiếp → no auth check. Dùng network segmentation hoặc mTLS giữa gateway và services.

USE CASE EXPERT
13.2 — API Composition & Response Aggregation
🔴 Problem Statement

Mobile app cần hiển thị dashboard: user profile + recent orders + recommendations + notifications. Nếu gọi 4 APIs riêng biệt → 4 round trips, mỗi round trip qua cellular network (200ms latency) = 800ms+ total. UX rất tệ trên mobile.

🟢 Solution

API Composition tại Gateway: Gateway expose /api/dashboard. Khi nhận request, gateway gọi song song 4 services (user, orders, recommendations, notifications), aggregate responses thành 1 JSON. Client chỉ cần 1 round trip = 200ms (thay vì 800ms). Gateway dùng goroutines/threads để fan-out requests. Nếu 1 service slow → timeout + partial response (dashboard vẫn hiển thị, thiếu 1 section).

📝 Note & Conclusion

Cẩn trọng: Gateway không nên chứa business logic. API composition chỉ nên là simple aggregation + formatting. Nếu logic phức tạp (ví dụ: filter orders by status, calculate totals) → tạo BFF service chuyên biệt.

14
Backend For Frontend (BFF)
API layer tối ưu cho từng loại client
EXPERT
💡 Định nghĩa
BFF là gì?

Backend For Frontend (BFF) là pattern tạo backend service riêng cho mỗi loại frontend client (web, mobile, IoT, partner API). Mỗi BFF chỉ expose data và format mà client đó cần, tối ưu cho characteristics cụ thể của từng platform (bandwidth, screen size, capabilities).

BFF giải quyết vấn đề "one API fits all": web app cần rich data (100 fields), mobile cần compact data (20 fields), IoT cần minimal data (5 fields). Một generic API không thể tối ưu cho tất cả.

Glossary chuyên sâu

BFF Layer
Thin backend layer giữa frontend và microservices. Xử lý data aggregation, transformation, formatting cụ thể cho client type.
API Composition
BFF gọi multiple downstream services, compose response tối ưu cho frontend. Ví dụ: mobile BFF chỉ lấy fields cần thiết.
Experience API
API thiết kế xung quanh user experience, không phải domain model. Mỗi endpoint = 1 screen/feature trên frontend.
GraphQL BFF
Dùng GraphQL thay vì REST cho BFF layer. Client tự specify fields cần → giảm over-fetching. Một BFF serve nhiều clients.
BFF Architecture — Per-Client Backend Services
USE CASE CƠ BẢN
14.1 — Mobile vs Web: Khác biệt Data Requirements
🔴 Problem Statement

Product detail page: Web app hiển thị 50 fields (full description, reviews, related products, seller history...), Mobile app chỉ hiển thị 15 fields (title, price, main image, rating). Dùng chung 1 API → mobile download 10x dữ liệu không cần thiết → tốn bandwidth, chậm trên 3G/4G.

🟢 Solution

Separate BFFs: Web BFF (/web/products/:id) trả full 50 fields, kèm HTML-friendly format. Mobile BFF (/mobile/products/:id) trả 15 fields, images ở resolution thấp hơn, payload gzip compressed. Mỗi BFF team align với frontend team tương ứng → develop, deploy independent. API contract tối ưu cho từng platform.

📝 Note & Conclusion

Trade-off: Thêm 1 service per client type = thêm code duplication (gọi cùng downstream services). Mitigation: share common libraries/SDK cho downstream calls. Hoặc dùng GraphQL làm unified BFF — client tự specify fields cần.

USE CASE EXPERT
14.2 — GraphQL BFF: Single BFF, Multiple Clients
🔴 Problem Statement

Công ty có Web, iOS, Android, Partner API — tạo 4 BFFs = too much overhead cho small team (5 backend developers). Nhưng vẫn cần tối ưu payload cho từng client.

🟢 Solution

GraphQL BFF: Tạo 1 GraphQL server đóng vai trò BFF cho tất cả clients. Schema define tất cả available data. Web query 50 fields, mobile query 15 fields — từ cùng schema. No over-fetching, no under-fetching. Kết hợp DataLoader pattern để batch + cache downstream service calls. 1 BFF, N clients, mỗi client tự tối ưu query cho mình.

📝 Note & Conclusion

Khi nào BFF vs GraphQL: BFF khi clients có logic khác nhau (web cần SSR, mobile cần push notification integration). GraphQL khi sự khác biệt chỉ là fields/payload. Cả hai có thể kết hợp: Mobile BFF dùng GraphQL internally.

Best Practices

  • BFF nên được own bởi frontend team (hoặc full-stack team). Backend team own downstream services.
  • BFF là thin layer: chỉ aggregation + transformation. Không chứa core business logic.
  • Mỗi BFF có thể dùng tech stack riêng: Web BFF dùng Node.js (SSR), Mobile BFF dùng Go (performance).
  • Dùng shared client libraries cho downstream service calls để tránh code duplication giữa BFFs.
  • BFF endpoints nên được thiết kế theo screen/feature, không theo domain model.
